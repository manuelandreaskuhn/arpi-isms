SIEM-System Name

Ein eindeutiger Name für Ihr SIEM-System.

Was ist ein SIEM?

Security Information and Event Management - das "Nervenzentrum" Ihrer IT-Sicherheit!

Warum SIEM essentiell ist:

  • Zentrale Log-Sammlung: Alle Logs an einem Ort
  • Angriffs-Erkennung: Korreliert Events über Systeme hinweg
  • Compliance: Audit-Trail für DSGVO, KRITIS, etc.
  • Incident Response: Schnelle Reaktion auf Sicherheitsvorfälle
  • Threat Hunting: Proaktive Suche nach Bedrohungen

Namens-Beispiele:

  • "SIEM-Prod" - Produktiv-SIEM
  • "Security-Monitoring" - Security Monitoring System
  • "SOC-SIEM" - SIEM für Security Operations Center
SIEM ist Pflicht für KRITIS und empfohlen für jedes Unternehmen ab 50+ Mitarbeiter!

SIEM-Software

Welche SIEM-Plattform wird verwendet?

SIEM-Kategorien im Detail:

  • Enterprise SIEM (Commercial):
    • Splunk Enterprise Security:
      • Marktführer mit ~30% Marktanteil
      • Beste Visualisierungen, Dashboards, Search-Language (SPL)
      • Über 1.000 Apps und Add-ons (Splunkbase)
      • Sehr flexibel, kann fast alles parsen
      • ABER: Sehr teuer (€100k-1M+ pro Jahr je nach GB/Tag)
      • Lizenz nach indexiertem Volume (GB/Tag)
      • Steep Learning Curve - aber mächtig wenn beherrscht
      • Gold-Standard in Security - "If you can afford it"
    • IBM QRadar:
      • Traditionell sehr stark in Enterprise
      • Exzellente Correlation-Engine
      • Lizenz nach Flows + Events per Second (FPS + EPS)
      • Gut für Compliance-Reports
      • Komplexe UI, steile Lernkurve
      • Integration mit IBM Security Portfolio (X-Force, Resilient)
      • Solid Enterprise-Wahl, besonders für IBM-Shops
    • Micro Focus ArcSight:
      • Einer der ältesten SIEMs (seit 2000)
      • Sehr mature, aber auch veraltet wirkend
      • Java-basiert, ressourcenhungrig
      • Gut für regulierte Industrien (Finance, Government)
      • Marktanteil sinkt (Splunk übernimmt)
      • Legacy-Option - nur wenn bereits vorhanden
    • Microsoft Sentinel (früher Azure Sentinel):
      • Cloud-native SIEM (nur Azure Cloud)
      • Tiefe Integration mit Microsoft 365, Azure, Windows
      • KQL (Kusto Query Language) - ähnlich SQL
      • Pay-as-you-go (per GB ingested)
      • Schnell einzurichten wenn Azure-Shop
      • ABER: Schwach bei Non-Microsoft-Sources
      • Perfekt für Microsoft-lastige Umgebungen
    • LogRhythm:
      • All-in-One SIEM (inkl. SOAR, NTA, UEBA)
      • Gute Performance
      • Einfacher als Splunk
      • Mittelständisch fokussiert
    • Securonix:
      • Fokus auf UEBA (User Behavior Analytics)
      • Cloud-native oder Hybrid
      • Machine Learning im Kern
  • Open Source SIEM:
    • Elastic Security (SIEM):
      • Basiert auf Elastic Stack (Elasticsearch + Kibana + Beats)
      • Open Source Core + Commercial Features (Platinum/Enterprise)
      • Sehr performant (Elasticsearch ist extrem schnell)
      • Flexibel - kann für Logs, Metrics, APM genutzt werden
      • Große Community, viele Integrations (Beats, Logstash)
      • Prebuilt Detection Rules (Elastic Security Rules)
      • MITRE ATT&CK Mapping integriert
      • Commercial: UEBA, Machine Learning, Case Management
      • Beste Open-Source-Option - Enterprise-Level kostenlos möglich!
    • Wazuh:
      • Fork von OSSEC, stark erweitert
      • Open Source Host-based IDS + SIEM + XDR
      • Komplett kostenlos (keine Commercial-Version)
      • Gut dokumentiert, aktive Community
      • Agenten für Windows, Linux, macOS
      • Integrations: VirusTotal, MISP, TheHive
      • Gut für kleinere bis mittlere Umgebungen
      • Perfekt für Einstieg ohne Budget!
    • OSSEC:
      • Original Open Source HIDS
      • Sehr leichtgewichtig
      • Weniger Features als Wazuh
      • Noch verwendet, aber Wazuh überholt
    • AlienVault OSSIM:
      • Open Source Version von AlienVault USM
      • All-in-One (SIEM, IDS, Asset Discovery, Vuln Scanning)
      • Gut für Startups
      • Limited Support
  • Cloud SIEM / SaaS:
    • Sumo Logic:
      • Cloud-native Log Analytics + SIEM
      • Pay-as-you-go
      • Gut für Cloud-Workloads (AWS, Azure, GCP)
    • Datadog Security:
      • Teil der Datadog Observability Platform
      • Fokus auf Cloud + APM + Security
      • Modern, aber teuer
    • Rapid7 InsightIDR:
      • Cloud SIEM + Incident Detection & Response
      • Einfach zu deployen
      • Gut für SMB

Entscheidungskriterien:

  • Budget:
    • Kein Budget: Wazuh, Elastic (Open Source)
    • Mittel (€50-100k): Elastic Security (Commercial), LogRhythm
    • Groß (€100k+): Splunk, QRadar
  • Größe:
    • <100 Devices: Wazuh, Elastic
    • 100-1000: Elastic, LogRhythm, Sentinel
    • 1000+: Splunk, QRadar
  • Expertise:
    • Wenig: Sentinel (wenn MS-Shop), Managed SIEM (MSSP)
    • Mittel: Elastic, LogRhythm
    • Hoch: Splunk (maximize value)
  • Cloud vs On-Premise:
    • Cloud-First: Sentinel, Sumo Logic
    • On-Prem: Splunk, QRadar, Elastic
    • Hybrid: Alle möglich
  • Compliance:
    • KRITIS, Healthcare: On-Prem bevorzugt (Splunk, Elastic)
    • PCI-DSS: Alle, aber Reports wichtig (QRadar gut)
Trend: Elastic gewinnt Marktanteile von Splunk (günstiger, open source)

Lizenzmodell

Wie wird das SIEM lizenziert?

Lizenz-Modelle erklärt:

  • Per EPS (Events per Second):
    • Z.B. IBM QRadar
    • 1.000 EPS, 5.000 EPS, 10.000 EPS
    • Überschreitung = Drosselung oder Mehrkosten
    • EPS schwer vorherzusagen!
  • Per GB/Tag:
    • Z.B. Splunk
    • 50 GB/Tag, 100 GB/Tag, 500 GB/Tag
    • Einfacher zu kalkulieren
    • Kann teuer werden bei viel Traffic
  • Per Device/Log Source:
    • Pro angeschlossenes System
    • Einfach zu kalkulieren
    • Unabhängig von Log-Volume
  • Per User: Anzahl SIEM-Benutzer
  • Subscription / SaaS: Monatlich/Jährlich, meist Cloud
  • Perpetual: Einmalkauf (selten bei SIEM)
  • Open Source / Kostenlos: Elastic, Wazuh (Support extra)
SIEM-Lizenzen können sehr teuer sein - gut kalkulieren!

Deployment-Typ

Wo läuft das SIEM?

Deployment-Optionen:

  • On-Premise:
    • Eigene Server im Rechenzentrum
    • Volle Kontrolle
    • Daten bleiben im Haus (DSGVO)
    • Wartung selbst machen
    • Standard für KRITIS
  • Cloud / SaaS:
    • Beim Anbieter gehostet
    • Schnelle Einrichtung
    • Kein Betrieb nötig
    • Pay-as-you-grow
    • Aber: Logs verlassen Organisation
  • Hybrid:
    • Log-Sammler On-Premise
    • Analyse in Cloud
    • Kompromiss
On-Premise bei sensiblen Daten (Patientendaten), Cloud bei kleinen Teams

Events per Second (EPS)

Wie viele Log-Events verarbeitet das SIEM pro Sekunde?

Was ist EPS?

Die "Durchsatzrate" Ihres SIEMs - wie viele Events können pro Sekunde aufgenommen, verarbeitet und indexiert werden.

EPS-Kategorisierung:

  • < 500 EPS: Small (Startup, <50 Mitarbeiter, <25 Server)
  • 500-2.000 EPS: Small-to-Medium (50-150 MA, 25-100 Server)
  • 2.000-10.000 EPS: Medium (150-500 MA, 100-500 Server)
  • 10.000-50.000 EPS: Large (500-2000 MA, 500-2000 Server)
  • >50.000 EPS: Enterprise (2000+ MA, 2000+ Server, Global)

Was generiert wie viel EPS:

  • Windows Server (Domain Controller): 50-200 EPS (sehr geschwätzig!)
  • Windows Client: 10-50 EPS
  • Linux Server: 5-20 EPS (weniger Logs als Windows)
  • Firewall: 100-1000 EPS (je nach Traffic)
  • Web Application: 50-500 EPS (je nach Load)
  • Database: 10-100 EPS
  • Network Device (Switch/Router): 5-50 EPS
  • Office 365: 20-200 EPS (per 100 User)

Berechnung der benötigten EPS:

Beispiel-Rechnung:
  • 10 Domain Controller × 100 EPS = 1.000 EPS
  • 50 Windows Server × 20 EPS = 1.000 EPS
  • 200 Windows Clients × 10 EPS = 2.000 EPS
  • 20 Linux Server × 10 EPS = 200 EPS
  • 2 Firewalls × 500 EPS = 1.000 EPS
  • 10 Applications × 50 EPS = 500 EPS
  • Summe: 5.700 EPS
Aber: Spitzen können 3-5x höher sein! → Lizenz für ~20.000 EPS sinnvoll

Peak vs. Average:

  • Average: Durchschnittliche Last über 24h
  • Peak: Spitzenlast (Montag 8 Uhr, nach Wochenende, Updates)
  • Peak kann 5-10x höher sein als Average!
  • SIEM muss Peak-Last bewältigen können

EPS-Lizenz zu klein → Probleme:

  • Logs werden gedrosselt (Queueing)
  • Events gehen verloren (Dropped Events)
  • Blind für Angriffe in Peak-Zeiten
  • Compliance-Violation (nicht alle Logs erfasst)

EPS vs. GB/Tag:

  • 1.000 EPS ≈ 50-100 GB/Tag (abhängig von Event-Größe)
  • 10.000 EPS ≈ 500-1000 GB/Tag
  • Windows Events: ~500 Bytes/Event
  • Firewall Logs: ~200 Bytes/Event

Best Practice:

  • Lizenz für 2-3x Average EPS (Peak-Reserve)
  • Monitoring der tatsächlichen EPS
  • Log-Filtering für unwichtige Events (Noise-Reduction)
Faustformel: 20-50 EPS pro aktiv loggendem Device (Windows höher, Linux niedriger)

Log-Quellen Typen

Welche Arten von Systemen senden Logs an das SIEM?

Wichtige Log-Quellen:

  • Firewalls: Blocked Connections, Allowed Traffic - essentiell!
  • IDS/IPS: Angriffs-Erkennungen
  • Server (Windows/Linux): System Events, Logins, Fehler
  • Active Directory: User Logins, Rechte-Änderungen - sehr wichtig!
  • Datenbanken: SQL-Queries, Zugriffe
  • Anwendungen: Applikations-Logs, Fehler
  • Netzwerkgeräte: Switches, Router
  • Endpoints (EDR): Antivirus, EDR-Alerts
  • Cloud Services: Office 365, AWS, Azure
  • E-Mail Security: Spam-Filter, Mail-Gateway

Je mehr Quellen, desto besser die Sichtbarkeit!

Use Cases / Detection Rules

Konfigurierte Erkennungsregeln für Sicherheitsvorfälle.

Was sind Use Cases?

Spezifische Erkennungsregeln die Log-Events korrelieren um Angriffe zu erkennen - wie "Rezepte" für Angriffs-Erkennung.

Use Case Kategorien mit Beispielen:

  • Authentication Attacks:
    • Brute-Force: 10 fehlgeschlagene Logins in 5 Min vom selben User
    • Password Spray: 1 Login-Versuch für 50+ Accounts in kurzer Zeit
    • Impossible Travel: User loggt sich aus USA und Deutschland in <1h ein
    • After-Hours Login: Admin-Login um 3 Uhr nachts
  • Lateral Movement:
    • Admin-Login von ungewöhnlichem System
    • PsExec-Nutzung zur Remote-Execution
    • SMB-Connections zu vielen Systemen (Worm-Verhalten)
  • Privilege Escalation:
    • User wird unerwartet zur Admin-Gruppe hinzugefügt
    • Neue lokale Admin-Accounts erstellt
    • Service-Account mit Admin-Rechten
  • Data Exfiltration:
    • Ungewöhnlich große Uploads zu Cloud-Storage
    • Hoher ausgehender Traffic zu unbekannten IPs
    • Große Datei-Downloads von Fileserver
  • Malware / C2 Communication:
    • Verbindung zu bekannter Command & Control IP
    • Beaconing (regelmäßige Verbindungen in festem Intervall)
    • DNS-Tunneling (ungewöhnlich lange DNS-Queries)
  • Insider Threats:
    • Zugriff auf ungewöhnliche Dateien/Datenbanken
    • Massenhafter Datei-Zugriff vor Kündigung
    • USB-Device-Nutzung (wenn verboten)
  • Policy Violations:
    • Shadow IT (unautorisierte Cloud-Services)
    • Firewall-Rule-Änderungen
    • Admin-Aktionen ohne Change-Ticket
  • Vulnerability Exploitation:
    • Erfolgreiche Exploit-Versuche (IDS-Signatur + Success)
    • SQL-Injection-Versuche
    • RCE (Remote Code Execution) Indicators

Use Case Reife-Level:

  • Level 1 (Basis, 20-50 Use Cases):
    • Out-of-the-Box Rules
    • Basic Attacks (Brute-Force, Malware-Kommunikation)
    • Viele False Positives
  • Level 2 (Mature, 50-150 Use Cases):
    • Getunete Rules (weniger False Positives)
    • Correlation über mehrere Log-Quellen
    • Custom Rules für eigene Umgebung
    • Ziel für die meisten Organisationen
  • Level 3 (Advanced, 150-500 Use Cases):
    • Threat Hunting Use Cases
    • Machine Learning / UEBA
    • Proaktive Detection (Indicators of Attack)
    • MITRE ATT&CK Coverage >80%
  • Level 4 (Elite, 500+ Use Cases):
    • Custom Detection für spezifische Threats
    • Threat Intelligence Integration
    • Behavioral Analytics
    • Red Team Emulation Detection

Use Case Development Prozess:

  1. Identify: Welche Angriffe wollen wir erkennen?
  2. Design: Welche Logs benötigen wir? Korrelations-Logik?
  3. Implement: Rule in SIEM schreiben
  4. Test: Erzeugt Rule Alerts? False Positives?
  5. Tune: Schwellwerte anpassen, Whitelists
  6. Deploy: Produktiv schalten
  7. Monitor: Alert-Volume beobachten
  8. Maintain: Regelmäßig reviewen und aktualisieren

Häufiger Fehler:

Zu viele Use Cases zu schnell aktivieren → Alert Fatigue → SOC überlastet → Wichtige Alerts werden übersehen

Empfehlung:

  • Start mit 20-30 High-Value Use Cases
  • Erst tunen bis False Positive Rate <10%
  • Dann schrittweise erweitern (5-10 neue Use Cases/Monat)
Qualität vor Quantität - 50 gut getunete Rules > 500 laut schreiende!

Machine Learning / UEBA

User and Entity Behavior Analytics - KI-gestützte Anomalie-Erkennung.

Was ist UEBA?

System das "normales" Verhalten von Usern und Entities (Systeme, IPs) lernt und Abweichungen erkennt - ohne vordefinierte Regeln.

Wie UEBA funktioniert:

  1. Baselining (2-4 Wochen):
    • Lernt normales Verhalten jedes Users
    • "User X loggt sich Mo-Fr 8-17 Uhr ein"
    • "User X greift auf Server A, B, C zu"
    • "User X lädt 50 MB/Tag herunter"
  2. Scoring:
    • Jede Aktion bekommt Risk-Score (0-100)
    • Je mehr Abweichung, desto höher Score
  3. Anomaly Detection:
    • Score > Schwellwert → Alert
    • Kombiniert mehrere Indikatoren

UEBA Use Cases (Beispiele):

  • Compromised Account:
    • User X loggt sich um 3 Uhr nachts ein (normal: 9-17 Uhr)
    • + greift auf Fileserver zu (macht er sonst nie)
    • + lädt 10 GB herunter (normal: 50 MB/Tag)
    • → High Risk Score → Alert
  • Insider Threat:
    • HR-Mitarbeiter greift plötzlich auf Engineering-Datenbank zu
    • Sales-Person lädt gesamte Kundenliste (macht er normalerweise nie)
  • Lateral Movement:
    • Service-Account loggt sich auf 50 Systemen ein (normal: 5 Systeme)
    • Muster: Angreifer verwendet kompromittierte Credentials
  • Data Exfiltration:
    • System kontaktiert plötzlich 100 neue externe IPs
    • Ungewöhnlich viel Upload zu Cloud-Storage
  • Malware Behavior:
    • Workstation öffnet plötzlich 1000+ Netzwerk-Connections
    • Regelmäßiges "Beaconing" (alle 60 Sekunden Verbindung → C2)

UEBA vs. Traditional Rules:

  • Traditional Rules:
    • Fest definiert: "Wenn X dann Alert"
    • Erkennt bekannte Angriffe
    • Braucht viel Tuning
    • False Positives häufig
  • UEBA:
    • Dynamisch: Lernt individuelles Normal-Verhalten
    • Erkennt Zero-Day und unbekannte Angriffe
    • Weniger Tuning nötig
    • Aber: Braucht Trainingszeit

Machine Learning Methoden in UEBA:

  • Unsupervised Learning: Clustering, Outlier-Detection
  • Supervised Learning: Classification (ist das Malware?)
  • Statistical Models: Bayes, Z-Score
  • Time-Series Analysis: Trend-Erkennung

Herausforderungen:

  • Training-Phase: 2-4 Wochen bis aussagekräftig
  • False Positives: Legitimate aber ungewöhnliches Verhalten
  • Concept Drift: Normal-Verhalten ändert sich (Home Office → plötzlich VPN normal)
  • Rechenleistung: ML-Modelle brauchen CPU/RAM
  • Erklärbarkeit: "Warum Alert?" oft schwer zu verstehen (Black Box)

Best Practices:

  • UEBA als Ergänzung zu Rules, nicht Ersatz
  • Start mit wenigen Entities (VIPs, Service-Accounts)
  • Kombination: UEBA findet Anomalie → Analyst untersucht mit traditionellen Tools
  • Regelmäßig Re-Training (Baseline anpassen)

UEBA-fähige SIEMs:

Splunk UBA, Microsoft Sentinel (Fusion), IBM QRadar UBA, Exabeam, Securonix
UEBA ist "next level" - sehr mächtig für Advanced Threats, aber komplex und braucht Expertise!

Threat Intelligence Integration

Integration externer Bedrohungs-Datenbanken.

Was ist Threat Intelligence?

Wissen über aktuelle Bedrohungen - "Steckbriefe" von Angreifern, Malware, C&C-Servern.

Threat Intel Quellen:

  • MISP: Open Source Threat Sharing Platform
  • AlienVault OTX: Open Threat Exchange (kostenlos)
  • Commercial Feeds: FireEye, CrowdStrike, Recorded Future (teuer, aber sehr aktuell)
  • Government: CERT-Bund, BSI, FBI Feeds

Was wird gematcht:

  • IP-Adressen (bekannte Angreifer-IPs)
  • Domains (C&C-Server, Phishing-Sites)
  • File-Hashes (bekannte Malware)
  • Indicators of Compromise (IoCs)

Beispiel:

Firewall-Log zeigt Verbindung zu IP 1.2.3.4 → SIEM matcht gegen Threat Intel → IP ist bekannter C&C-Server → High-Priority Alert!
Threat Intel macht SIEM von reaktiv zu proaktiv!

MITRE ATT&CK Framework

Standard-Framework zur Kategorisierung von Angriffstechniken.

Was ist MITRE ATT&CK?

Eine weltweit anerkannte "Enzyklopädie" aller bekannten Cyber-Angriffstechniken - das "Playbook" aus dem Hacker lesen.

Geschichte:

Erstellt von MITRE Corporation (Non-Profit, US-Government-Auftragnehmer) seit 2013. Wird kontinuierlich aktualisiert basierend auf realen Angriffen.

Struktur im Detail:

  • Tactics (Taktiken) - Das "Warum":
    • 14 Taktiken: Die strategischen Ziele eines Angreifers
    • Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion, Credential Access, Discovery, Lateral Movement, Collection, Command & Control, Exfiltration, Impact
  • Techniques (Techniken) - Das "Wie":
    • ~200 Techniken: Konkrete Methoden
    • Z.B. T1078: Valid Accounts, T1059: Command-Line Interface
  • Sub-Techniques:
    • ~600+ Sub-Techniken: Noch spezifischer
    • Z.B. T1078.001: Default Accounts, T1078.002: Domain Accounts
  • Procedures:
    • Wie spezifische Threat-Actors die Technik nutzen
    • Real-World-Beispiele

Die 14 Tactics (Attack Lifecycle):

  1. Initial Access: Wie kommt Angreifer rein? (Phishing, Exploit)
  2. Execution: Code ausführen (PowerShell, Scripts)
  3. Persistence: Nach Reboot noch da (Autostart, Service)
  4. Privilege Escalation: Admin-Rechte erlangen
  5. Defense Evasion: Nicht entdeckt werden (AV umgehen)
  6. Credential Access: Passwörter stehlen (Mimikatz)
  7. Discovery: Netzwerk erkunden (Port-Scan)
  8. Lateral Movement: Zu anderen Systemen springen
  9. Collection: Daten sammeln (Screenshots, Keylogger)
  10. Command & Control (C2): Mit Angreifer-Server kommunizieren
  11. Exfiltration: Daten rausschmuggeln
  12. Impact: Schaden anrichten (Ransomware, Löschen)

MITRE im SIEM - Praktische Nutzung:

  • Use Case Mapping:
    • Jeder Detection Rule MITRE Technique zuordnen
    • "Brute-Force Detection" → T1110: Brute Force
    • "PsExec Lateral Movement" → T1021.002: SMB/Windows Admin Shares
  • Coverage-Matrix:
    • Visualisierung: Welche Techniques werden erkannt?
    • Gap-Analysis: Wo sind Lücken?
    • Tool: MITRE ATT&CK Navigator (https://mitre-attack.github.io/attack-navigator/)
  • Incident Investigation:
    • Alert kommt rein → zu welcher Tactic/Technique gehört das?
    • Nächste Schritte vorhersagen: "Nach T1078 kommt meist T1069 (Discovery)"
  • Threat Intel Integration:
    • APT-Gruppe nutzt Techniques X, Y, Z
    • Detection Rules für genau diese Techniques priorisieren

MITRE ATT&CK Matrices:

  • Enterprise: Windows, Linux, macOS, Cloud (AWS, Azure, GCP)
  • Mobile: Android, iOS
  • ICS: Industrial Control Systems (SCADA, etc.)

Andere Detection Frameworks:

  • Cyber Kill Chain (Lockheed Martin):
    • Älter als MITRE (2011)
    • 7 Phasen: Recon → Weaponization → Delivery → Exploitation → Installation → C2 → Actions
    • Einfacher, aber weniger detailliert
  • NIST Cybersecurity Framework:
    • 5 Funktionen: Identify → Protect → Detect → Respond → Recover
    • Mehr Compliance-fokussiert, weniger technisch
  • Sigma Rules:
    • Open Signature Format für SIEM Rules
    • SIEM-agnostisch (convertierbar zu Splunk, Elastic, QRadar)
    • Community-driven (GitHub)
    • Oft mit MITRE-Tags versehen

MITRE in Action - Beispiel:

APT29 (Cozy Bear / Russian SVR):
  • Initial Access: T1566.002 (Spearphishing Link)
  • Execution: T1059.001 (PowerShell)
  • Persistence: T1053.005 (Scheduled Task)
  • Defense Evasion: T1070.004 (File Deletion)
  • Credential Access: T1003 (Credential Dumping)
  • Discovery: T1082 (System Information Discovery)
  • Lateral Movement: T1021.001 (Remote Desktop Protocol)
  • Collection: T1005 (Data from Local System)
  • Exfiltration: T1041 (Exfiltration Over C2 Channel)
→ SOC kann Use Cases für genau diese Techniques aktivieren!
MITRE ATT&CK ist DIE gemeinsame Sprache im Cyber Security - jedes SOC sollte es nutzen!

Log-Retention (Aufbewahrung)

Wie lange werden Logs gespeichert und in welchen Tiers?

Compliance-Anforderungen im Detail:

  • DSGVO (EU):
    • Speicherdauer-Minimierung: "Nicht länger als nötig"
    • Typisch 6-12 Monate für Security-Logs
    • Personenbezogene Daten nach Zweckerfüllung löschen
    • Aber: Sicherheits-Incidents rechtfertigen längere Speicherung
  • KRITIS (DE - BSI-Gesetz):
    • Minimum 6 Monate für Angriffs-Erkennung
    • Empfohlen: 12 Monate
    • Incident-relevante Logs: 7 Jahre
  • PCI DSS (Payment Card Industry):
    • Requirement 10.7: Minimum 90 Tage online verfügbar
    • Minimum 1 Jahr Archiv (schnell abrufbar)
    • Audit-Logs müssen manipulationssicher sein
  • SOX (Sarbanes-Oxley Act):
    • 7 Jahre für finanzrelevante Logs
    • E-Mails, Transaktionen, Audit-Trails
  • HIPAA (US Healthcare):
    • 6 Jahre für Healthcare-relevante Logs
  • NIS2-Richtlinie (EU, ab 2024):
    • Kritische Infrastruktur: Logs für Forensik aufbewahren
    • Empfohlen: 12-24 Monate

Storage-Tier-Strategie (Best Practice):

  • Hot Storage (Online / Fast):
    • 30-90 Tage
    • SSD / High-Performance Storage
    • Sofortige Search möglich (<1 Sekunde)
    • Für aktives Threat Hunting und Incident Response
    • Teuer: ~€1-5 per GB/Monat
  • Warm Storage (Nearline / Medium Speed):
    • 90 Tage - 1 Jahr
    • HDD / Standard Storage
    • Search in Minuten möglich
    • Für Compliance-Reports und historische Analyse
    • Medium: ~€0.10-0.50 per GB/Monat
  • Cold Storage (Archiv / Slow):
    • 1-7 Jahre
    • Tape, Object Storage (S3 Glacier), Archiv-Systeme
    • Search in Stunden/Tagen (Restore nötig)
    • Nur für Compliance und lange Aufbewahrung
    • Günstig: ~€0.01-0.05 per GB/Monat

Beispiel-Retention-Policy:

  • Security-kritische Logs (Firewall, AD, IDS):
    • 90 Tage Hot
    • 1 Jahr Warm
    • 7 Jahre Cold
  • Standard-Server-Logs:
    • 30 Tage Hot
    • 6 Monate Warm
    • 1 Jahr Cold (dann löschen)
  • Application Debug-Logs:
    • 7 Tage Hot (dann löschen - zu viel Noise)
  • Compliance-relevante Logs (Financial, Healthcare):
    • 90 Tage Hot
    • 7 Jahre Cold (Compliance-Pflicht)

Storage-Berechnung (Beispiel):

Szenario: 500 GB Logs/Tag
  • Hot (90 Tage): 500 GB × 90 = 45 TB × €2/GB = €90k/Monat
  • Warm (1 Jahr): 500 GB × 365 = 182 TB × €0.30/GB = €54k/Monat
  • Cold (7 Jahre, komprimiert 10:1): 500 GB × 365 × 7 ÷ 10 = 127 TB × €0.02/GB = €2.5k/Monat
  • Gesamt: ~€146k/Monat Storage-Kosten!
→ Daher: Log-Filtering wichtig (nur relevante Logs aufheben)

Log-Lifecycle-Management:

  1. Logs werden ingested → Hot Storage
  2. Nach 90 Tagen: Automatisch zu Warm Storage verschieben
  3. Nach 1 Jahr: Komprimieren + zu Cold Storage
  4. Nach 7 Jahren: Automatisch löschen (außer Incident-relevant)

Rechtliche Aspekte:

  • Legal Hold: Bei laufendem Rechtsverfahren dürfen Logs NICHT gelöscht werden
  • Deletion Policy: Nach Ablauf Retention automatisch löschen (DSGVO-Konform)
  • Audit-Trail: Wer hat wann welche Logs gelöscht? (Manipulationsschutz)

Cloud vs. On-Premise Storage:

  • Cloud (AWS S3, Azure Blob):
    • Unlimitiert skalierbar
    • Tier-System integriert (S3 Standard → Glacier)
    • Pay-as-you-go
    • Aber: Egress-Kosten beim Abrufen!
  • On-Premise:
    • Hardware-Investition nötig
    • Volle Kontrolle
    • Gut für sensible Daten (KRITIS)
Retention-Strategie: Security-Bedarf + Compliance + Kosten in Balance bringen!

Alerting & Notification

Wie werden Sicherheitsvorfälle kommuniziert?

Alert-Channels:

  • E-Mail: Standard, aber langsam
  • SMS: Für kritische Alerts (Kosten!)
  • Slack / MS Teams: Modern, schnell, Team-Kollaboration
  • Webhook: Integration mit anderen Tools
  • SIEM Dashboard: Zentrale Übersicht

Alert-Fatigue Problem:

  • Zu viele Alerts = werden ignoriert
  • Typisch: 90%+ False Positives am Anfang
  • Tuning nötig: Schwellwerte anpassen, Whitelists

Alert-Priorisierung:

  • Critical: SMS + Anruf (z.B. Ransomware)
  • High: Slack + Ticket (z.B. Brute-Force)
  • Medium: Nur Dashboard (z.B. Policy-Violation)
  • Low: Nur Logging (z.B. Information)
Gutes SIEM: 10-50 Alerts/Tag, nicht 1000!

SOC (Security Operations Center)

Das Team, das das SIEM überwacht.

Was ist ein SOC?

Das "Kontrollzentrum" für IT-Sicherheit - Analysten überwachen SIEM 24/7.

SOC-Typen:

  • Internes SOC:
    • Eigene Mitarbeiter
    • Im eigenen Rechenzentrum
    • Volle Kontrolle
    • Teuer (3-4 FTE für 24/7)
  • Externes SOC / MSSP:
    • Managed Security Service Provider
    • Outsourcing der Überwachung
    • 24/7 ohne eigenes Personal
    • Günstiger, aber weniger Kontrolle
  • Hybrid:
    • MSSP für 24/7-Überwachung
    • Internes Team für Tier-2/3 und Incident Response
    • Guter Kompromiss

SOC-Betriebszeiten:

  • 24/7: Für kritische Infrastruktur (KRITIS) - ideal
  • 8x5: Nur Geschäftszeiten - günstiger, aber Lücken
  • Extended Hours: Z.B. 6-22 Uhr - Kompromiss
Ohne SOC ist SIEM nur teures Log-Storage - jemand muss die Alerts bearbeiten!

Notizen / SIEM-Besonderheiten

Zusätzliche wichtige Informationen zum SIEM-System.

Was dokumentieren:

  • Spezielle Use Cases und Custom Rules
  • Tuning-Historie (welche Regeln wurden angepasst)
  • Bekannte False Positives und Whitelists
  • Integration-Besonderheiten (Probleme mit bestimmten Log-Quellen)
  • Performance-Probleme und Lösungen
  • Geplante Erweiterungen (neue Log-Quellen)
  • SOC-Prozesse und Playbooks
  • Eskalations-Pfade

Beispiel:

"Splunk Enterprise Security 7.1 on-premise. 150 Log-Quellen, ~3.500 EPS Average, Peaks 8.000 EPS. 180 Tage Hot Storage (15 TB), 2 Jahre Cold (Tape). 127 aktive Use Cases, Coverage 85% MITRE ATT&CK Tactics. UEBA aktiviert für AD-Accounts. Threat Intel: MISP + AlienVault OTX + FireEye (Commercial). SOAR: Splunk Phantom mit 23 Playbooks. Bekannte Issues: Windows DNS Logs erzeugen viele False Positives → gefiltert nach Whitelist. O365-Integration über MS Graph API (Rate-Limit beachten!). SOC: Hybrid - MSSP für L1 24/7 (SecureWorks), internes Team für L2/L3 (8x5). Kritische Alerts: Slack #security-alerts + SMS an On-Call. Eskalation: CISO bei Critical Incidents. Backup: Täglich nach S3, Tests quartalsweise."
SIEM-Dokumentation ist essentiell - bei Incident keine Zeit zum Rätseln!
