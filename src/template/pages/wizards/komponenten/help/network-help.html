Netzwerk-Name

Ein eindeutiger, beschreibender Name für das Netzwerk-Segment.

Was ist ein Netzwerk-Segment?

Ein logisch oder physisch abgetrennter Teil des Netzwerks - wie "Räume" in einem Haus.

Warum Netzwerk-Segmentierung wichtig ist:

  • Sicherheit: Trennung verschiedener Bereiche (z.B. Produktion von Gäste-WLAN)
  • Performance: Reduziert Broadcast-Traffic
  • Compliance: DSGVO/KRITIS verlangen Segmentierung
  • Troubleshooting: Probleme einfacher lokalisieren

Namens-Beispiele:

  • "Prod-Network" - Produktiv-Netzwerk
  • "VLAN-100-KIS" - VLAN 100 für KIS
  • "DMZ-Webserver" - DMZ für Webserver
  • "Management-Network" - Management-Netzwerk
Gute Namen = schnelleres Verständnis der Netzwerk-Architektur!

Netzwerk-Typ

Die Art bzw. Funktion des Netzwerks.

Netzwerk-Typen erklärt:

  • LAN (Local Area Network):
    • Lokales Netzwerk in einem Gebäude/Standort
    • Typisch: Büro-Netzwerk, flache Struktur
    • Hohe Geschwindigkeit (1-100 Gbit/s)
    • Geringe Latenz (<1ms)
  • VLAN (Virtual LAN):
    • Logisch getrenntes Netzwerk auf gleicher Hardware
    • Segmentierung ohne zusätzliche physische Switches
    • Flexibel und kosteneffizient
    • Broadcast-Domain-Isolation
    • Standard im modernen Netzwerk!
  • WAN (Wide Area Network):
    • Verbindung zwischen geografisch getrennten Standorten
    • Über Internet, MPLS oder dedizierte Leitungen
    • Langsamer als LAN (typisch 10-1000 Mbit/s)
    • Höhere Latenz (10-100ms+)
  • DMZ (Demilitarized Zone):
    • Pufferzone zwischen Internet und internem Netz
    • Für öffentlich erreichbare Server (Web, Mail, DNS)
    • Wenn kompromittiert: Internes Netz bleibt geschützt
    • Firewall-Regeln: Extern→DMZ (restriktiv), DMZ→Intern (sehr restriktiv)
  • Management Network:
    • Für Administration von Servern, Switches, Storage
    • Out-of-Band-Management (separates physisches Netz)
    • Getrennt vom Produktiv-Traffic
    • Nur für Admins/Monitoring erreichbar
    • Typisch: IPMI, iLO, iDRAC, SSH, SNMP
  • Storage Network (SAN):
    • Dediziert für Storage-Traffic (iSCSI, NFS, SMB)
    • Hohe Bandbreite erforderlich (10G/25G/100G)
    • Niedrige Latenz kritisch
    • Von Produktiv-Netz getrennt (Performance + Security)
    • Jumbo Frames (MTU 9000) oft aktiviert
  • WiFi / WLAN:
    • Drahtloses Netzwerk (802.11ac/ax)
    • Oft mehrere SSIDs → mehrere VLANs
    • Security: WPA3-Enterprise mit 802.1X
  • VPN Network:
    • Verschlüsseltes Overlay-Netzwerk
    • Für Remote-Zugriff oder Site-to-Site
    • Eigener IP-Bereich (z.B. 10.8.0.0/24)
Verschiedene Netzwerk-Typen haben unterschiedliche Security- und Performance-Anforderungen!

VLAN-ID

Die eindeutige Nummer des VLANs (1-4094).

Was ist eine VLAN-ID?

Eine Nummer (Tag), die Netzwerk-Pakete einem VLAN zuordnet.

Typische VLAN-Bereiche:

  • VLAN 1: Default VLAN (nicht nutzen!)
  • VLAN 10-99: Management, Infrastruktur
  • VLAN 100-199: Server, Produktion
  • VLAN 200-299: Clients, Arbeitsplätze
  • VLAN 300-399: Gast, IoT
  • VLAN 400-499: DMZ, öffentliche Server

Best Practice:

Dokumentieren Sie Ihr VLAN-Schema - z.B. "100er = Server, 200er = Clients"
VLAN 1 niemals produktiv nutzen - Sicherheitsrisiko!

Netzwerk-Adresse (CIDR)

Die IP-Adresse und Größe des Netzwerks in CIDR-Notation.

Was ist CIDR?

Classless Inter-Domain Routing - kompakte Schreibweise für Netzwerke. Ersetzt alte Klassen-basierte Adressierung (Class A/B/C).

Format:

Netzwerk-IP/Präfixlänge (z.B. 192.168.100.0/24)

Häufige Präfixlängen:

  • /30: 4 IPs (2 nutzbar) - Point-to-Point Links
  • /29: 8 IPs (6 nutzbar) - Sehr kleine Netze
  • /28: 16 IPs (14 nutzbar) - Kleine Servergruppe
  • /27: 32 IPs (30 nutzbar)
  • /26: 64 IPs (62 nutzbar)
  • /25: 128 IPs (126 nutzbar)
  • /24: 256 IPs (254 nutzbar) - Typisch für Abteilungs-Netze
  • /23: 512 IPs (510 nutzbar)
  • /22: 1.024 IPs (1.022 nutzbar)
  • /21: 2.048 IPs
  • /20: 4.096 IPs
  • /16: 65.536 IPs - Große Unternehmen
  • /8: 16 Millionen IPs - Sehr große Provider

Wichtig:

Erste IP = Netzwerk-Adresse (nicht nutzbar)
Letzte IP = Broadcast-Adresse (nicht nutzbar)
→ Bei /24: 256 IPs, aber nur 254 nutzbar!

Private IP-Bereiche (RFC 1918):

  • 10.0.0.0/8 (10.0.0.0 - 10.255.255.255) - 16 Mio. IPs
  • 172.16.0.0/12 (172.16.0.0 - 172.31.255.255) - 1 Mio. IPs
  • 192.168.0.0/16 (192.168.0.0 - 192.168.255.255) - 65k IPs

Subnetting-Faustregel:

Immer Wachstum einplanen - lieber zu groß als zu klein!
CIDR-Rechner nutzen für komplexes Subnetting: ipcalc, sipcalc, subnet-calculator.com

Gateway / Router

Die IP-Adresse des Standard-Gateways (meist der Router).

Was ist ein Gateway?

Der "Ausgang" aus dem Netzwerk - alle Pakete zu anderen Netzen gehen durchs Gateway.

Typische Gateway-IPs:

  • Erste IP: 192.168.100.1 (häufigste Konvention)
  • Letzte IP: 192.168.100.254
  • Zweite IP: 192.168.100.2

Beispiel:

Netzwerk 192.168.100.0/24 → Gateway meist 192.168.100.1
Gateway ist meist erste oder letzte nutzbare IP des Netzes

DHCP (Dynamic Host Configuration Protocol)

Automatische IP-Vergabe an Geräte.

Was ist DHCP?

Server vergibt automatisch IPs an neue Geräte - wie "Zimmervergabe" im Hotel.

DHCP vergibt nicht nur IP, sondern auch:

  • Subnetzmaske
  • Default Gateway
  • DNS-Server
  • NTP-Server (optional)
  • TFTP-Server (für PXE-Boot, optional)
  • Domain-Name (optional)

DHCP vs. Statische IPs:

  • DHCP (Dynamisch):
    • IP wird automatisch vergeben
    • Einfach für Clients (PCs, Laptops, Phones)
    • IP kann sich ändern (nach Lease-Ablauf)
    • Zentrale Verwaltung
    • Standard für Endgeräte
  • Statisch:
    • IP manuell am Gerät konfiguriert
    • IP ändert sich nie
    • Für Server, Drucker, Netzwerkgeräte, Gateways
    • Aufwändiger zu verwalten
    • Wichtig für Infrastruktur!
  • DHCP-Reservierung (Best of both):
    • Gerät nutzt DHCP, bekommt aber immer gleiche IP
    • Basiert auf MAC-Adresse
    • Zentral verwaltbar + stabile IP
    • Ideal für Drucker, IoT-Geräte

DHCP-Sicherheit:

DHCP Snooping aktivieren gegen Rogue DHCP-Server!
Faustregel: Server = statisch, Clients = DHCP, Drucker/IoT = DHCP-Reservierung

DHCP-Pool (IP-Range)

Der Bereich von IPs, die DHCP vergeben darf.

IP-Adress-Planung - Best Practice:

Netzwerk in Bereiche aufteilen für bessere Organisation

Beispiel-Planung für 192.168.100.0/24 (256 IPs):

  • .1-.1: Gateway/Router (192.168.100.1)
  • .2-.9: Netzwerk-Infrastruktur (Switches, APs)
  • .10-.49: Server (statisch zugewiesen)
  • .50-.99: Drucker, IoT (DHCP-Reservierungen)
  • .100-.200: DHCP-Pool für Clients (dynamisch)
  • .201-.249: Reserve für Wachstum
  • .250-.254: Management/Test/temporäre Geräte

Alternative kompakte Planung:

  • .1 = Gateway
  • .2-50 = Statische IPs (Infrastruktur + Server)
  • .51-200 = DHCP-Pool (Clients)
  • .201-254 = Reserve

Wichtig:

  • DHCP-Pool darf NICHT das ganze Netz umfassen
  • Genug statische IPs für Infrastruktur reservieren
  • Reserve für zukünftiges Wachstum einplanen
  • Dokumentieren welcher Bereich für was genutzt wird
Gut geplante IP-Ranges erleichtern Troubleshooting enorm!

DHCP-Lease-Zeit

Wie lange eine IP-Adresse "gemietet" wird bevor Erneuerung nötig ist.

Typische Lease-Zeiten:

  • 8 Stunden (28800s):
    • Für Umgebungen mit häufigen Geräte-Wechseln
    • Gast-Netzwerke, Schulen, Messen
  • 24 Stunden (86400s):
    • Standard für die meisten Büro-Netze
    • Guter Kompromiss aus Flexibilität und Stabilität
  • 3-7 Tage:
    • Für stabile Umgebungen (Home Office VPN-Clients)
    • Weniger DHCP-Traffic
  • Unbegrenzt (infinite):
    • Quasi statisch, aber zentral verwaltet
    • Selten verwendet (unflexibel)

Lease-Erneuerung:

Client versucht bei 50% der Lease-Zeit zu erneuern (T1)
Falls fehlgeschlagen: erneut bei 87.5% (T2)
→ Praktisch sehen Clients ihre IP lange konstant

Kurze Lease-Zeit:

  • ✓ Mehr Flexibilität
  • ✓ IPs werden schneller frei
  • ✗ Mehr DHCP-Server-Last
  • ✗ Mehr Netzwerk-Traffic

Lange Lease-Zeit:

  • ✓ Weniger Last auf DHCP-Server
  • ✓ Stabilere IP-Zuordnung
  • ✗ IPs länger gebunden (auch wenn Gerät weg)
  • ✗ Pool-Erschöpfung möglich

Empfehlung:

Standard = 24h, außer spezielle Anforderungen

Sicherheitszone

Die Vertrauensstufe dieses Netzwerks.

Zonen-Konzept:

Netzwerke nach Vertrauenslevel trennen

Sicherheitszonen erklärt:

  • Internal / Trusted:
    • Internes Produktiv-Netz
    • Höchste Vertrauensstufe
    • Für geschäftskritische Systeme
  • DMZ:
    • Pufferzone für öffentliche Server
    • Mittleres Vertrauen
    • Wenn kompromittiert: Internes Netz geschützt
  • External / Untrusted:
    • Internet, externe Netzwerke
    • Kein Vertrauen
    • Alles blockieren außer explizit erlaubt
  • Management Zone:
    • Nur für Admins
    • Isoliert vom Produktiv-Netz
  • Guest Network:
    • Für Gäste/Besucher
    • Nur Internet, kein interner Zugriff
  • Restricted / High Security:
    • Sehr sensible Daten (z.B. Patientendaten)
    • Strenge Zugriffskontrolle

Netzwerk-Isolation

Wie stark ist dieses Netz von anderen getrennt?

Isolations-Level:

  • Vollständig isoliert:
    • Kein Routing zu anderen Netzen
    • Air-Gap (physisch getrennt)
    • Für sehr kritische Systeme
  • Teilweise isoliert:
    • Routing über Firewall mit strengen Regeln
    • Nur spezifische Verbindungen erlaubt
    • Standard für Produktiv-Netze
  • Keine Isolation:
    • Freies Routing zwischen Netzen
    • Nur für unkritische Bereiche
Mehr Isolation = höhere Sicherheit, aber komplexere Verwaltung

802.1X (Port-based NAC)

Authentifizierung bevor Netzwerkzugang gewährt wird.

Was ist 802.1X?

Standard für Port-basierte Netzwerk-Zugriffskontrolle (Network Access Control) - wie "Ausweiskontrolle" am Netzwerk-Port.

Komponenten:

  • Supplicant: Client (PC, Laptop) - will ins Netz
  • Authenticator: Switch/AP - kontrolliert Port
  • Authentication Server: RADIUS-Server - entscheidet über Zugang

Wie funktioniert's:

  1. Gerät verbindet sich mit Switch-Port
  2. Port ist blockiert (nur EAP-Traffic erlaubt)
  3. Gerät authentifiziert sich:
    • EAP-TLS: Mit Zertifikat (sicherste Methode)
    • PEAP: Username/Passwort in verschlüsseltem Tunnel
    • EAP-TTLS: Ähnlich PEAP, flexibler
  4. RADIUS-Server prüft Berechtigung (Active Directory, LDAP)
  5. Bei Erfolg: Port wird freigeschaltet
  6. Switch weist Gerät dynamisch richtiges VLAN zu

Vorteile:

  • Nur autorisierte Geräte/User im Netz
  • Dynamische VLAN-Zuweisung (Role-based Access)
  • Schutz vor Rogue Devices (unauthorisierte Geräte)
  • Zentrale Verwaltung über RADIUS
  • Audit-Trail (wer wann wo im Netz)

Herausforderungen:

  • Komplex zu konfigurieren (Switch + RADIUS + Zertifikate)
  • Nicht alle Geräte unterstützen 802.1X:
    • IoT-Geräte (oft keine 802.1X-Unterstützung)
    • Legacy-Geräte
    • Drucker (teilweise)
    • → Lösung: MAC-Authentication-Bypass (MAB)
  • Braucht RADIUS-Infrastruktur
  • Zertifikats-Management (bei EAP-TLS)

MAB (MAC Authentication Bypass):

Fallback für Geräte ohne 802.1X - Switch authentifiziert mit MAC-Adresse beim RADIUS
802.1X + EAP-TLS ist Gold-Standard für Enterprise-Security!

MAC-Adress-Filtering

Zugriffskontrolle basierend auf Hardware-Adressen.

Was ist MAC-Filtering?

Nur Geräte mit bekannter MAC-Adresse dürfen ins Netz.

Methoden:

  • Whitelist: Nur gelistete MACs erlaubt (restriktiv)
  • Blacklist: Gelistete MACs blockiert (permissiv)

Problem:

MAC-Adressen sind fälschbar (MAC-Spoofing) → keine echte Sicherheit!

Besser:

802.1X mit Zertifikaten
MAC-Filtering = Security durch Obskurität, nicht wirklich sicher!

DHCP Snooping

Schutz vor Rogue DHCP-Servern.

Das Problem ohne DHCP Snooping:

Angreifer bringt eigenen DHCP-Server ins Netz →
Verteilt falsche IP-Konfiguration (z.B. Angreifer als Gateway) →
Man-in-the-Middle: Aller Client-Traffic läuft über Angreifer!

Was ist DHCP Snooping?

Switch-Feature das lernt, welche Ports "vertrauenswürdig" sind (zu echtem DHCP-Server führen).

Wie's funktioniert:

  • Trusted Ports:
    • DHCP-Server-Antworten (OFFER, ACK) erlaubt
    • Typisch: Uplink-Ports zum Core/DHCP-Server
  • Untrusted Ports:
    • DHCP-Server-Antworten blockiert
    • DHCP-Client-Anfragen (DISCOVER, REQUEST) erlaubt
    • Typisch: Alle Access-Ports zu Clients
  • Binding-Tabelle:
    • Switch protokolliert: IP ↔ MAC ↔ Port ↔ VLAN ↔ Lease-Time
    • Wird für DAI und IP Source Guard verwendet

Zusätzliche Schutz-Features:

  • Rate Limiting: Max. DHCP-Pakete/Sekunde pro Port (gegen DoS)
  • Option 82: Switch fügt Informationen zu DHCP-Paketen hinzu (Port-ID, VLAN)

Basis für andere Security-Features:

DHCP Snooping muss aktiviert sein für:
  • Dynamic ARP Inspection (DAI)
  • IP Source Guard

Konfiguration (Cisco-Beispiel):

ip dhcp snooping
ip dhcp snooping vlan 100
interface GigabitEthernet0/1 → trusted
interface range GigabitEthernet0/2-24 → untrusted
DHCP Snooping ist Layer-2-Security-Grundlage - immer aktivieren!

Dynamic ARP Inspection (DAI)

Schutz vor ARP-Spoofing/Poisoning-Angriffen.

Was ist ARP (Address Resolution Protocol)?

Übersetzt IP-Adresse → MAC-Adresse
Beispiel: PC will zu 192.168.1.1 → sendet ARP-Request "Wer hat 192.168.1.1?" → Gateway antwortet mit seiner MAC

ARP-Spoofing/Poisoning Angriff:

  1. Angreifer sendet falsche ARP-Antworten
  2. "192.168.1.1 (Gateway) hat MAC-Adresse von Angreifer!"
  3. Clients aktualisieren ARP-Cache mit falscher Info
  4. Clients senden Traffic für Gateway an Angreifer
  5. Angreifer kann mitlesen/manipulieren (Man-in-the-Middle)
  6. Angreifer leitet Traffic an echtes Gateway weiter (bleibt unbemerkt)

Was ist DAI?

Switch prüft ARP-Pakete gegen DHCP-Snooping-Binding-Tabelle.

Wie's funktioniert:

  • Switch vergleicht ARP-Paket mit Binding-Tabelle:
    • Sender-IP ↔ Sender-MAC ↔ Port
  • Stimmt nicht überein → ARP-Paket wird gedroppt
  • Log-Eintrag erstellt

Beispiel-Szenario:

  • Legitim: 192.168.1.100 (MAC: aa:bb:cc, Port 10) sendet ARP → DAI: OK ✓
  • Angriff: Port 15 sendet ARP für 192.168.1.1 → Nicht in Binding-Table → DAI: BLOCK ✗

Trusted vs. Untrusted Ports:

  • Trusted: Keine DAI-Prüfung (Uplinks, zu Routern)
  • Untrusted: Volle DAI-Prüfung (Client-Ports)

ARP ACLs:

Für statische IPs (nicht in DHCP-Binding-Table) müssen ARP ACLs definiert werden

Voraussetzung:

  • DHCP Snooping muss aktiviert und Binding-Table gefüllt sein
  • Oder: ARP ACLs für statische IP-Geräte

Rate Limiting:

Schützt Switch-CPU vor ARP-Storm (max. ARP-Pakete/Sekunde)
DAI verhindert effektiv ARP-Man-in-the-Middle-Angriffe!

STP/RSTP/MSTP (Spanning Tree Protocol)

Verhindert Switch-Loops und ermöglicht redundante Pfade.

Das Problem ohne STP:

  • Redundante Switch-Verbindungen → Layer-2-Loop
  • Broadcast-Pakete laufen im Kreis (werden vervielfacht)
  • Broadcast-Storm → alle Switches/Clients überlastet
  • Netzwerk komplett down in Sekunden!

Was ist STP:

Protokoll das automatisch redundante Pfade erkennt und blockiert, aber bei Ausfall aktiviert → Loop-frei + redundant!

Versionen:

  • STP (802.1d) - Classic Spanning Tree:
    • Original-Standard von 1990
    • Konvergenz: 30-50 Sekunden (sehr langsam!)
    • Max. 7 Hops
    • Nur ein Spanning Tree für alle VLANs
    • Veraltet - nicht mehr nutzen!
  • RSTP (802.1w) - Rapid Spanning Tree:
    • Verbesserung von STP (seit 2001)
    • Konvergenz: 1-6 Sekunden (sehr schnell!)
    • Abwärtskompatibel zu STP
    • Neue Port-Rollen (Alternate, Backup)
    • Standard für die meisten Netze heute
  • MSTP (802.1s) - Multiple Spanning Tree:
    • Mehrere Spanning Trees (Instanzen)
    • Pro VLAN-Gruppe ein separater Tree
    • Bessere Load-Balancing-Möglichkeiten
    • Komplex zu konfigurieren
    • Für große, VLAN-intensive Netze

Wie STP funktioniert (vereinfacht):

  1. Root Bridge wählen: Switch mit niedrigster Priority (oder MAC) wird "Chef"
  2. Root Ports bestimmen: Jeder Switch wählt besten Port zur Root Bridge
  3. Designated Ports: Pro Segment ein Port zum Forwarden
  4. Blocking Ports: Alle anderen Ports blockieren (Loop-Prevention)
  5. Bei Ausfall: Blockierter Port wird aktiviert (Failover)

Port-States (RSTP):

  • Discarding: Port blockiert (keine Frames)
  • Learning: Lernt MAC-Adressen (kein Forwarding)
  • Forwarding: Normal aktiv (Traffic fließt)

STP-Optimierungen:

  • PortFast: Port sofort in Forwarding (für Clients, NIE zwischen Switches!)
  • BPDU Guard: Port disable bei STP-Frame-Empfang (Schutz vor Rogue Switches)
  • Root Guard: Verhindert dass Port Root Bridge wird
  • Loop Guard: Verhindert Loops durch einseitige Link-Failures

Best Practices:

  • RSTP oder MSTP nutzen (nicht STP)
  • Root Bridge manuell setzen (nicht zufällig)
  • Backup Root Bridge definieren
  • PortFast + BPDU Guard auf allen Access-Ports
RSTP ist absolute Pflicht bei redundanten Switch-Topologien!

Link Aggregation (LACP)

Mehrere Netzwerk-Links zu einem logischen bündeln.

Was ist LACP?

Link Aggregation Control Protocol (802.3ad) - mehrere Kabel = ein schnellerer Link.

Vorteile:

  • Höhere Bandbreite: 2x 1G = 2G, 4x 10G = 40G
  • Redundanz: Ein Kabel ausfällt → Rest läuft weiter
  • Load Balancing: Traffic verteilt auf alle Links

Beispiel:

Server mit 4x 10G Netzwerkkarten → LACP → 40G Bandbreite zum Switch

Varianten:

  • LACP (dynamic): Automatische Aushandlung
  • Static Port Channel: Manuell konfiguriert
LACP ist Standard für Server-Uplinks!

VRRP/HSRP/GLBP (Gateway Redundancy)

Redundante Default-Gateways für Hochverfügbarkeit.

Das Problem:

Client hat Default Gateway 192.168.1.1 (Router A)
Router A fällt aus → Client kann nicht mehr ins Internet/andere Netze
Auch wenn Router B vorhanden ist - Client kennt ihn nicht!

Die Lösung: First Hop Redundancy Protocols (FHRP)

Mehrere Router teilen sich eine virtuelle IP-Adresse

Protokolle im Detail:

  • VRRP (Virtual Router Redundancy Protocol):
    • Open Standard (RFC 5798)
    • Hersteller-unabhängig (Cisco, Juniper, HP, etc.)
    • Active-Standby (nur ein Router aktiv)
    • Master-Router: Priority 255-1 (höher = besser)
    • Preemption: Standard aktiviert
    • Multicast: 224.0.0.18
    • Empfohlen für Multi-Vendor-Umgebungen
  • HSRP (Hot Standby Router Protocol):
    • Cisco-proprietär (aber am weitesten verbreitet)
    • Active-Standby
    • Active-Router: Priority 255-0 (höher = besser, Default: 100)
    • Preemption: Optional (muss aktiviert werden)
    • Multicast: 224.0.0.2 (v1), 224.0.0.102 (v2)
    • Hello-Timer: 3 Sekunden, Hold: 10 Sekunden
    • Standard in Cisco-Umgebungen
  • GLBP (Gateway Load Balancing Protocol):
    • Cisco-proprietär
    • Active-Active (alle Router aktiv!)
    • Load Balancing über mehrere Router
    • AVG (Active Virtual Gateway) + bis zu 4 AVFs (Active Virtual Forwarders)
    • Verschiedene LB-Methoden: Round-Robin, Weighted, Host-dependent
    • Bessere Ressourcen-Nutzung als HSRP/VRRP
    • Ideal für Umgebungen mit hohem Traffic

Wie's funktioniert (HSRP-Beispiel):

  • Setup:
    • Router A: Physical IP 192.168.1.2, Priority 110
    • Router B: Physical IP 192.168.1.3, Priority 100
    • Virtual IP: 192.168.1.1 (das Default Gateway)
  • Normal-Betrieb:
    • Router A = Active (weil höhere Priority)
    • Router A antwortet auf ARP für 192.168.1.1
    • Router A routet allen Traffic
    • Router B = Standby (Heartbeat-Überwachung)
  • Failover:
    • Router A fällt aus (keine Heartbeats mehr)
    • Nach 10 Sekunden (Hold-Timer): Router B übernimmt
    • Router B sendet gratuitous ARP: "Ich bin jetzt 192.168.1.1"
    • Clients merken nichts (benutzen weiter 192.168.1.1)
    • Downtime: ~10 Sekunden (anpassbar)
  • Failback (mit Preemption):
    • Router A kommt zurück
    • Router A übernimmt wieder (höhere Priority)

Tracking:

Priority automatisch reduzieren wenn bestimmte Schnittstelle/Route ausfällt
→ Automatisches Failover auch bei Teil-Ausfällen

Best Practices:

  • Preemption aktivieren (automatisches Failback)
  • Authentication aktivieren (MD5)
  • Tracking für kritische Links konfigurieren
  • Timer anpassen (schnelleres Failover bei Bedarf)
Gateway-Redundanz ist absolute Pflicht für produktive Netze!

QoS (Quality of Service)

Priorisierung und Verwaltung von Netzwerk-Traffic.

Warum QoS nötig:

Nicht aller Traffic ist gleich wichtig und zeitkritisch!

Das Problem ohne QoS:

  • VoIP-Anruf stockt/Gesprächsabbruch, weil großer File-Download läuft
  • Video-Konferenz pixeliert bei Backup-Traffic
  • Geschäftskritische Anwendung langsam durch unkritischen Traffic

Traffic-Charakteristiken:

  • VoIP:
    • Latenz-sensitiv (<150ms)
    • Jitter-sensitiv (<30ms)
    • Bandbreite: gering (64-128 Kbit/s pro Call)
    • → Höchste Priorität!
  • Video:
    • Latenz-sensitiv
    • Burst-Traffic
    • Bandbreite: hoch (1-10 Mbit/s)
  • Geschäftsanwendungen:
    • Transaktional (ERP, Datenbank)
    • Moderate Priorität
  • Best Effort (Standard):
    • Web, E-Mail
    • Keine Garantien
  • Scavenger (niedrig):
    • Backup, File-Sync, Updates
    • Darf andere nicht stören

QoS-Mechanismen:

  • Classification & Marking:
    • Traffic erkennen und markieren
    • Layer 2: CoS (802.1p) - 3 Bit = 8 Prioritäten (0-7)
    • Layer 3: DSCP (DiffServ) - 6 Bit = 64 Klassen
  • Queuing:
    • Traffic in verschiedene Warteschlangen sortieren
    • Priority Queuing (PQ): Höchste Queue zuerst (Starvation-Gefahr!)
    • Weighted Fair Queuing (WFQ): Fair aber priorisiert
    • Class-Based WFQ (CBWFQ): Bandbreiten-Garantien pro Klasse
  • Policing:
    • Traffic über Limit: Droppen oder Remarken
    • Hart (sofortiges Droppen)
    • Einfach zu implementieren
  • Shaping:
    • Traffic über Limit: Puffern und verzögern
    • Weich (Traffic wird geglättet)
    • Braucht Buffer
  • Congestion Avoidance:
    • WRED (Weighted Random Early Detection)
    • Droppen vor totaler Congestion

DSCP-Werte (wichtigste):

  • EF (Expedited Forwarding, 46): VoIP, echtzeitkritisch
  • AF41 (34): Video, hohe Priorität
  • AF31 (26): Geschäftskritische Anwendungen
  • AF21 (18): Standard-Anwendungen
  • CS6 (48): Netzwerk-Management (Routing-Protokolle)
  • Best Effort (0): Standard (kein QoS)
  • CS1 (8): Scavenger/Bulk (niedrigste Priorität)

CoS zu DSCP Mapping (typisch):

  • CoS 7 → CS7 (56) - Network Control
  • CoS 6 → CS6 (48) - Internetwork Control
  • CoS 5 → EF (46) - VoIP
  • CoS 4 → AF41 (34) - Video
  • CoS 3 → AF31 (26) - Business-Critical
  • CoS 0-2 → Best Effort

QoS-Modelle:

  • Best Effort: Kein QoS (Standard Internet)
  • IntServ: Pro-Flow Reservation (RSVP) - komplex, selten
  • DiffServ: Klassen-basiert - Standard heute!

Trust-Boundary:

Wo werden QoS-Markierungen vertrauenswürdig akzeptiert?
Typisch: An Access-Ports neu klassifizieren (Clients können lügen!)
QoS ist essentiell für VoIP, Video und Unified Communications!

NetFlow / sFlow / IPFIX

Monitoring von Netzwerk-Traffic-Flows.

Was ist ein Flow:

Eine Verbindung zwischen zwei Endpoints (z.B. PC → Webserver).

Flow-Export-Protokolle:

  • NetFlow:
    • Cisco-Standard (auch von anderen unterstützt)
    • Detaillierte Flow-Informationen
    • Versions: v5 (alt), v9 (flexibel)
  • sFlow:
    • Sampling-basiert (nur Teil des Traffics)
    • Geringere Switch-Last
    • Weniger genau
  • IPFIX:
    • IETF-Standard (Nachfolger von NetFlow v9)
    • Sehr flexibel

Warum Flow-Monitoring wichtig:

  • Wer kommuniziert mit wem?
  • Welche Anwendungen verursachen Traffic?
  • Anomalie-Erkennung (DDoS, Data Exfiltration)
  • Kapazitätsplanung
  • Troubleshooting

Collector-Software:

ntopng, ElastiFlow, Plixer Scrutinizer
Flow-Monitoring = "Wer spricht mit wem im Netzwerk"

VPN (Virtual Private Network)

Verschlüsselte Netzwerk-Verbindung.

VPN-Typen:

  • Site-to-Site VPN:
    • Verbindet zwei Standorte
    • Transparent für Endbenutzer
    • Router/Firewall baut Tunnel auf
    • Beispiel: Hauptsitz ↔ Zweigstelle
  • Remote Access VPN:
    • Einzelne Benutzer verbinden sich
    • VPN-Client auf Laptop/PC nötig
    • Für Home Office, Dienstreisen

VPN-Protokolle:

  • IPSec: Standard für Site-to-Site, sehr sicher
  • SSL VPN: Browser-basiert, einfach
  • OpenVPN: Open Source, flexibel
  • WireGuard: Modern, sehr schnell, einfach

Compliance-Anforderungen

Welche gesetzlichen/regulatorischen Standards müssen erfüllt werden?

Standards erklärt:

  • PCI DSS:
    • Payment Card Industry Data Security Standard
    • Für Netzwerke die Kreditkarten verarbeiten
    • Verlangt: Segmentierung, Firewall, Logging
  • HIPAA:
    • Health Insurance Portability and Accountability Act (USA)
    • Für Gesundheitsdaten
    • Verschlüsselung, Zugriffskontrolle, Audit-Logs
  • DSGVO:
    • Datenschutz-Grundverordnung (EU)
    • Für alle personenbezogenen Daten
    • Segmentierung, Verschlüsselung, Logging
  • KRITIS:
    • Kritische Infrastrukturen (DE)
    • Krankenhäuser, Energie, Wasser
    • BSI-Grundschutz, ISO 27001
    • Netzwerk-Segmentierung Pflicht!
Compliance ist nicht optional - bei Verstößen drohen Millionen-Bußgelder!

IP-Adress-Management (IPAM)

Tool zur Verwaltung von IP-Adressen.

Warum IPAM wichtig:

  • Übersicht: Welche IPs sind belegt?
  • Planung: Wachstum, neue Netze
  • Vermeidung von IP-Konflikten
  • Dokumentation

IPAM-Lösungen:

  • phpIPAM: Open Source, Web-basiert, beliebt
  • NetBox: Open Source, auch für DCIM
  • Infoblox: Commercial, Enterprise
  • Microsoft IPAM: Teil von Windows Server
  • Excel: Für kleine Umgebungen (nicht empfohlen bei Wachstum!)
Excel ist KEIN IPAM - ab 100+ IPs besser echtes Tool nutzen!

Notizen / Netzwerk-Besonderheiten

Zusätzliche wichtige Informationen zu diesem Netzwerk.

Was dokumentieren:

  • Spezielle Routing-Regeln
  • Bekannte Probleme und Workarounds
  • Geplante Änderungen (z.B. "Migration auf neues VLAN geplant Q2/2025")
  • Historische Informationen (warum so konfiguriert?)
  • Performance-Bottlenecks
  • Abhängigkeiten zu anderen Netzen
  • Wartungsfenster

Beispiel:

"VLAN 100 für KIS-Server. Firewall-Regel FW-PROD-001 erlaubt nur Port 1433 (SQL) von VLAN 200. QoS-Profile 'Database-Priority' aktiv. Bekanntes Problem: Bei Lastspitzen > 80% Switch-CPU gelegentlich Packet Loss → Upgrade auf 10G geplant Q3/2025. Wartungsfenster: Sonntag 02:00-06:00 Uhr. Netzwerk-Diagramm: \\fileserver\netdocs\VLAN100.vsdx"
Gute Netzwerk-Dokumentation ist Gold wert - spart Stunden bei Troubleshooting!
