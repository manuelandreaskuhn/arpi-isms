VPN-Name

Ein eindeutiger Name für Ihre VPN-Verbindung.

Was ist ein VPN?

Virtual Private Network - ein verschlüsselter, sicherer Tunnel durch unsichere Netzwerke (meist das Internet). Wie ein "Unterwasser-Tunnel" unter einem See - niemand sieht was drin transportiert wird.

Warum VPN essentiell ist:

  • Sicherer Remote-Zugriff:
    • Home Office sicher ins Firmennetz
    • Außendienst von unterwegs
    • Zugriff wie im Büro
  • Standort-Verbindung (Site-to-Site):
    • Zweigstellen sicher mit Hauptstandort verbinden
    • Transparent für Benutzer
    • Günstiger als dedizierte Leitung
  • Verschlüsselung:
    • Daten geschützt über Internet übertragen
    • Auch über unsichere WLANs (Hotel, Café)
    • Abhörschutz
  • Zugriff auf interne Ressourcen:
    • Fileserver, Drucker, Intranet
    • Als wäre man physisch im Büro
  • Schutz sensibler Daten:
    • DSGVO-Compliance bei Remote-Arbeit
    • Schutz vor Man-in-the-Middle-Angriffen

Namens-Beispiele (Best Practices):

  • "Corporate-VPN" - Haupt-VPN für alle Mitarbeiter
  • "Site-to-Site-Berlin" - Permanente Verbindung zum Standort Berlin
  • "Admin-VPN" - Separates VPN nur für IT-Admins (erhöhte Sicherheit)
  • "Partner-VPN-SupplierX" - Dediziertes VPN für externen Geschäftspartner
  • "VPN-Developers" - VPN für Entwickler-Team (eigene ACLs)
  • "AWS-VPN-eu-central-1" - VPN zur AWS-Region

Kosten (Richtwerte):

  • VPN-Gateway (Hardware): €2.000-€20.000 (je nach Kapazität)
  • Lizenzen: €50-€200 pro gleichzeitiger Verbindung/Jahr
  • Cloud VPN: €0,05-€0,15 pro VPN-Connection-Stunde
  • Open Source (OpenVPN): Kostenlos (aber Betrieb selbst)
VPN ist essentiell für sicheres mobiles Arbeiten - ohne VPN = Sicherheitsrisiko!

VPN-Typ

Die grundlegende Architektur der VPN-Verbindung.

VPN-Typen im Detail:

  • Remote Access VPN (Client-to-Site):
    • Was: Einzelne Benutzer verbinden sich zum Firmennetz
    • Wie: VPN-Client-Software auf Laptop/PC installiert
    • Wann: User klickt "Verbinden"
    • Use Cases:
      • Home Office (häufigstes Szenario)
      • Außendienst (Vertrieb unterwegs)
      • Geschäftsreisen
      • Contractor/Freelancer-Zugang
    • Typisch: 10-1000 gleichzeitige Verbindungen
    • 90% aller Unternehmens-VPNs sind Remote Access!
  • Site-to-Site VPN (LAN-to-LAN):
    • Was: Verbindet zwei ganze Netzwerke/Standorte
    • Wie: VPN-Gateway an beiden Enden (Firewall/Router)
    • Wann: Permanent verbunden (24/7)
    • Transparent: Endbenutzer merken nichts vom VPN
    • Use Cases:
      • Hauptsitz ↔ Zweigstelle
      • Büro ↔ Rechenzentrum
      • On-Premise ↔ Cloud (Hybrid)
      • Partner-Anbindung (B2B)
    • Vorteil: Günstiger als dedizierte Leitung (MPLS)
    • Nachteil: Abhängig von Internet-Qualität
  • Mobile VPN:
    • Besonderheit: Hält VPN-Verbindung bei Netzwechsel
    • Szenario: Smartphone wechselt von WLAN zu 4G/5G
    • Normale VPNs: Verbindung bricht ab → neu verbinden
    • Mobile VPN: Nahtloser Übergang ohne Abbruch
    • Use Cases:
      • Mobile Workforce (Techniker, Außendienst)
      • Tablets/Smartphones
      • Kritische Anwendungen (dürfen nicht abbrechen)
    • Technologie: Meist IPsec IKEv2 mit MOBIKE
  • SSL VPN / Web VPN:
    • Was: Zugriff über Browser (HTTPS)
    • Kein Client nötig: Nur Browser
    • Port: 443 (wie normale HTTPS-Seite)
    • Use Cases:
      • Partner/Lieferanten (wollen keinen Client installieren)
      • Kiosk-PCs/öffentliche Computer
      • Schneller Zugriff ohne Installation
      • Zugriff auf Web-Apps (Intranet, Webmail)
    • Einschränkung: Meist nur Web-Anwendungen
    • Vorteil: Funktioniert überall (Firewall-friendly)
  • Clientless VPN / Portal:
    • Noch einfacher: Nur Web-Portal
    • Zero Footprint: Nichts wird installiert
    • Zugriff: Nur auf Web-basierte Ressourcen
    • Technologie: Reverse Proxy im Browser
    • Use Cases:
      • Sehr eingeschränkter Zugriff
      • Partner mit minimalen Rechten
      • Quick Access zu einzelnen Web-Apps

VPN-Protokoll

Das verwendete Verschlüsselungs- und Tunneling-Protokoll.

VPN-Protokolle erklärt:

  • IPsec (IKEv2):
    • Moderner Standard
    • Sehr sicher
    • Gut für mobile Geräte (reconnect)
    • Empfohlen!
  • IPsec (IKEv1):
    • Ältere Version
    • Noch weit verbreitet
    • Weniger Features als IKEv2
  • SSL/TLS VPN:
    • Browser-basiert möglich
    • Läuft über HTTPS (Port 443)
    • Gut durch Firewalls
  • OpenVPN:
    • TCP oder UDP möglich
    • Sehr flexibel
    • Open Source
  • WireGuard:
    • Sehr modern, sehr schnell
    • Einfacher Code (weniger Bugs)
    • Zukunft des VPN
  • L2TP/IPsec:
    • Kombination aus L2TP + IPsec
    • Gut für ältere Systeme
    • Windows-nativ unterstützt
  • PPTP:
    • ALT und UNSICHER!
    • NICHT mehr verwenden
    • Nur für Legacy-Systeme
IPsec IKEv2 oder WireGuard = moderne Wahl, PPTP = niemals!

Verschlüsselungs-Algorithmus

Der Algorithmus zum Verschlüsseln der VPN-Daten.

Warum Verschlüsselung kritisch ist:

VPN ohne starke Verschlüsselung = nutzlos! Daten können abgehört werden.

Verschlüsselungs-Algorithmen im Detail:

  • AES-256-GCM (Recommended!):
    • AES-256: Advanced Encryption Standard, 256-bit Schlüssel
    • GCM: Galois/Counter Mode (AEAD - Authenticated Encryption with Associated Data)
    • Sicherheit: Militär-Grade (NSA Suite B)
    • Performance: Hardware-beschleunigt (AES-NI auf modernen CPUs)
    • Authentifizierung: Eingebaut (verhindert Manipulation)
    • Geschwindigkeit: Multi-Gbit/s möglich
    • Best Practice für alle VPNs!
  • AES-128-GCM:
    • 128-bit Schlüssel (statt 256)
    • Auch sehr sicher (2^128 Möglichkeiten = praktisch unknackbar)
    • Etwas schneller als 256 (marginal)
    • Ausreichend für die meisten Use Cases
    • Wenn Performance kritisch ist
  • AES-256-CBC / AES-128-CBC:
    • CBC: Cipher Block Chaining (älterer Modus)
    • Keine eingebaute Authentifizierung (braucht separates HMAC)
    • Langsamer als GCM
    • Anfällig für Padding Oracle Attacks (wenn falsch implementiert)
    • Legacy - nur verwenden wenn GCM nicht verfügbar
  • ChaCha20-Poly1305:
    • Alternative zu AES (von Google/D.J. Bernstein)
    • Vorteil: Schnell auch OHNE Hardware-AES (z.B. Smartphones, IoT)
    • Moderne Kryptografie (2014)
    • Von WireGuard verwendet
    • Sicherheit: Gleichwertig zu AES-256
    • Gut für Geräte ohne AES-Hardware-Beschleunigung
  • 3DES (Triple DES) - VERALTET!
    • Ur-alt (1970er DES, 3x ausgeführt)
    • Nur 112-bit effektive Sicherheit (schwach!)
    • Sehr langsam
    • Seit 2019 offiziell deprecated (NIST)
    • NIEMALS für neue VPNs verwenden!
    • Nur für uralte Legacy-Systeme die kein AES können

Performance-Vergleich (typisch auf moderner CPU):

  • AES-256-GCM: ~10 Gbit/s
  • AES-128-GCM: ~12 Gbit/s
  • ChaCha20-Poly1305: ~5 Gbit/s (ohne AES-NI)
  • AES-256-CBC: ~3 Gbit/s
  • 3DES: ~100 Mbit/s (50x langsamer!)

Empfehlung:

  • Standard: AES-256-GCM
  • Wenn sehr alte Hardware: AES-128-GCM
  • Mobile/IoT: ChaCha20-Poly1305
  • Niemals: 3DES (außer absolut keine andere Wahl)
AES-256-GCM = sicher, schnell, modern - 3DES = veraltet, langsam, unsicher!

Authentifizierungs-Methode

Wie authentifizieren sich Benutzer beim VPN?

Auth-Methoden erklärt:

  • Zertifikat-basiert:
    • Digitales Zertifikat auf Client
    • Sehr sicher (2-Faktor: haben + wissen)
    • Keine Passwörter nötig
    • Verwaltung aufwendiger
    • Best Practice für Site-to-Site!
  • Benutzername/Passwort:
    • Einfachste Methode
    • Benutzer kennen es
    • Aber: Passwörter können kompromittiert werden
    • MFA unbedingt dazu!
  • Zertifikat + Passwort:
    • Kombination: Sehr sicher
    • Haben (Zertifikat) + Wissen (Passwort)
    • Etwas komplexer für Benutzer
  • Multi-Faktor (MFA):
    • Z.B. Passwort + SMS/App-Code
    • Sehr sicher gegen Phishing
    • Heute Standard!
    • Empfohlen für alle Remote-Access-VPNs
  • Pre-Shared Key (PSK):
    • Gemeinsames Geheimnis
    • Einfach, aber: ein PSK für alle
    • Nur für Site-to-Site OK
    • Nie für Benutzer-VPNs!
  • Smartcard / PKI:
    • Physische Chipkarte
    • Höchste Sicherheit
    • Teuer in Anschaffung/Verwaltung
MFA ist heute Pflicht - ohne = Sicherheitsrisiko!

Routing-Modus

Welcher Traffic geht durchs VPN?

Routing-Modi erklärt:

  • Full Tunnel (gesamter Traffic):
    • ALLE Internet-Verbindungen durch VPN
    • Auch YouTube, Netflix, etc.
    • Höchste Sicherheit
    • Aber: Mehr Last auf VPN
    • Best Practice für Compliance!
  • Split Tunnel (nur Firmen-Traffic):
    • Nur internes Firmen-Netz durch VPN
    • Internet direkt (ohne VPN)
    • Weniger Last auf VPN
    • Schneller für Benutzer
    • Aber: Benutzer nicht vollständig geschützt
  • Inverse Split Tunnel:
    • Alles durch VPN AUSSER bestimmte Sites
    • Z.B. Office 365 direkt
    • Kompromiss zwischen Full und Split

Beispiel:

  • Full Tunnel: Mitarbeiter surft Google → durch Firmen-VPN → Firmen-Internet
  • Split Tunnel: Mitarbeiter surft Google → direkt über sein privates WLAN
Full Tunnel = sicherer, Split Tunnel = schneller

Endpoint Compliance Check (Posture Check / NAC)

Prüfung des Sicherheitszustands des Client-Geräts vor VPN-Zugang.

Was ist Posture Check / Network Access Control?

VPN-Gateway prüft Sicherheitszustand des Clients BEVOR Zugang zum Firmennetz gewährt wird. Wie ein "Gesundheitscheck" an der Tür.

Was wird typischerweise geprüft:

  • Antivirus / Endpoint Protection:
    • Ist installiert? (z.B. Microsoft Defender, CrowdStrike)
    • Ist aktiviert (nicht deaktiviert)?
    • Ist aktuell (Definitions-Stand < 7 Tage)?
    • Letzter Scan? (z.B. < 24h)
  • Personal Firewall:
    • Windows Firewall aktiviert?
    • macOS Firewall aktiviert?
  • OS Patch-Level:
    • Windows: Alle kritischen Updates installiert?
    • macOS: Aktuelle OS-Version? (z.B. nicht älter als 3 Versionen)
    • Bekannte kritische CVEs gepatcht?
  • Festplatten-Verschlüsselung:
    • BitLocker (Windows) aktiviert?
    • FileVault (macOS) aktiviert?
    • Wichtig bei Laptop-Verlust!
  • Verbotene Software:
    • Torrent-Clients (Copyright-Risiko)
    • TeamViewer/AnyDesk (unautorisierter Remote-Zugriff)
    • Hacking-Tools (Wireshark, Metasploit - je nach Policy)
  • Compliance-Software:
    • Ist Mobile Device Management (MDM) Client installiert?
    • Ist Endpoint Detection & Response (EDR) aktiv?

Aktionen bei fehlgeschlagenem Check:

  • Zugang komplett verweigert (Strict Mode):
    • Kein VPN-Zugang
    • Fehler-Meldung: "Ihr Gerät erfüllt nicht die Sicherheitsanforderungen"
    • Höchste Sicherheit, aber User-Frustration
  • Eingeschränkter Zugang (Quarantine Network):
    • VPN-Zugang nur zu Update-Servern
    • User kann Windows Update/Antivirus-Update laden
    • Nach erfolgreicher Remediation → voller Zugang
    • Praktischer Kompromiss
  • Warnung + Zugang (Soft Fail):
    • Warnung anzeigen, aber Zugang gewähren
    • IT-Team wird informiert
    • User muss innerhalb X Tagen nachbessern
    • Für Übergangsphase
  • Nur Logging (Monitor Mode):
    • Prüfung läuft, aber keine Aktion
    • Nur für Testing/Rollout

Technologie:

  • Agent-basiert: VPN-Client prüft selbst (z.B. Cisco AnyConnect Posture Module)
  • Agentless: VPN-Gateway scannt Client (z.B. SNMP, WMI)
  • Hybrid: Kombination

Herausforderungen:

  • False Positives: Legitime Software wird als "verboten" erkannt
  • User-Frustration: "Ich kann nicht arbeiten, weil ein Update fehlt!"
  • Maintenance-Overhead: Policies müssen gepflegt werden
  • BYOD-Problem: Private Geräte schwer zu kontrollieren

Best Practices:

  • Start mit Soft Fail (Warnung), dann schrittweise verschärfen
  • Clear Communication an User (warum wird abgelehnt, wie fixen?)
  • Self-Service-Portal: "Dein PC ist nicht compliant, hier klicken für Updates"
  • Ausnahme-Prozess: Manager kann temporär Zugang freischalten
  • Regelmäßig Policy reviewen (nicht zu strikt, nicht zu lasch)

Compliance-Standards:

  • PCI DSS: Posture Check empfohlen
  • HIPAA: Verschlüsselung Pflicht
  • DSGVO: Schutz personenbezogener Daten → Posture Check sinnvoll
Posture Check = wichtige Security-Schicht, aber braucht gute Kommunikation und User-Support!

Kill Switch / Drop-on-Disconnect

Blockiert automatisch Internet-Zugang wenn VPN-Verbindung abbricht.

Was ist ein Kill Switch?

Sicherheits-Feature das Internet sofort blockiert wenn VPN-Verbindung verloren geht - verhindert "ungeschütztes Surfen".

Das Problem OHNE Kill Switch (Szenario):

  1. Mitarbeiter verbindet sich mit VPN von Café-WLAN
  2. Arbeitet an sensitiven Dokumenten (Patientendaten, Finanzen)
  3. VPN bricht ab (schlechtes WLAN, Verbindungsproblem, Server-Neustart)
  4. Computer reconnected automatisch - DIREKT ins Café-WLAN (ohne VPN!)
  5. Mitarbeiter merkt es nicht (App läuft weiter)
  6. PROBLEM: Sensitive Daten gehen jetzt unverschlüsselt über unsicheres WLAN!
  7. Kann von jedem im Café mitgehört werden (Packet Sniffing)

Mit Kill Switch (wie es funktioniert):

  1. VPN-Client überwacht Verbindung kontinuierlich
  2. VPN bricht ab
  3. SOFORT: Kill Switch aktiviert sich
  4. Alle Netzwerk-Interfaces werden geblockt (außer loopback)
  5. Firewall-Regeln: Blockiere ALLES außer VPN-Server
  6. User sieht: "Keine Internet-Verbindung" + Warnung "VPN getrennt"
  7. User MUSS VPN neu verbinden um Internet zu nutzen
  8. Keine Möglichkeit für ungeschützte Verbindung

Technische Implementation:

  • Windows: Windows Filtering Platform (WFP) Rules
  • macOS: Packet Filter (pf) oder Network Extension
  • Linux: iptables / nftables Rules
Regeln: 
# Allow only VPN-Server
ALLOW: dst=vpn.company.com port=443
ALLOW: dst=vpn.company.com port=500,4500

# Allow local traffic
ALLOW: dst=127.0.0.0/8

# Block everything else
DENY: ALL

Modi:

  • Always-On Kill Switch:
    • Blockiert Internet auch wenn VPN NIE verbunden war
    • Sehr strikt
    • Nur Internet über VPN möglich
  • On-Disconnect Kill Switch:
    • Blockiert nur wenn VPN war verbunden und jetzt getrennt
    • Erlaubt direktes Internet wenn VPN nie gestartet
    • Häufigste Variante
  • Application-Level Kill Switch:
    • Blockiert nur bestimmte Apps (z.B. Browser, E-Mail)
    • Andere Apps (z.B. Spotify) dürfen weiter

Vorteile:

  • ✓ Verhindert unbeabsichtigten Daten-Leak
  • ✓ Schutz vor MITM-Angriffen
  • ✓ Compliance (DSGVO, HIPAA)
  • ✓ User kann nicht "versehentlich" ungeschützt surfen

Nachteile:

  • ✗ Bei VPN-Problemen = KEIN Internet (Frustration)
  • ✗ Debugging schwieriger (kann man VPN-Server erreichen?)
  • ✗ False Positives: Kurze Disconnects → Internet weg
  • ✗ Technischer User kann umgehen (Admin-Rechte → Firewall-Regeln löschen)

Wann Kill Switch essentiell:

  • Full-Tunnel VPN (gesamter Traffic durch VPN)
  • Sensitive Daten (DSGVO, Healthcare, Finance)
  • Untrusted Networks (öffentliche WLANs, Hotel, Café)
  • Compliance-Anforderungen

Wann Kill Switch optional:

  • Split-Tunnel VPN (nur Firmen-Traffic durch VPN)
  • Trusted Network (z.B. nur von Home Office)
  • Nicht-sensitive Anwendungen

Best Practice:

  • Kill Switch = ON für Full-Tunnel
  • Kill Switch = OFF für Split-Tunnel (sonst bei VPN-Ausfall auch privates Internet weg)
  • Clear User-Kommunikation: "Wenn VPN weg, dann Internet blockiert - das ist Schutz!"
  • Escape-Hatch: Hotline-Nummer wenn User nicht weiterkommt
Kill Switch = Best Practice für Full-Tunnel-VPNs mit sensitiven Daten!

Always-On VPN

VPN baut sich automatisch auf ohne User-Interaktion.

Was ist Always-On VPN?

VPN startet automatisch beim Booten des Computers und bleibt permanent verbunden - User sieht/macht nichts.

Wie Always-On funktioniert:

  1. Computer startet (oder Wake from Sleep)
  2. VPN-Client startet AUTOMATISCH (als System-Service)
  3. VPN verbindet BEVOR User einloggt (Pre-Logon)
  4. User loggt sich ein → ist bereits im VPN
  5. Bei Netzwerk-Wechsel (WLAN ↔ LTE) → Auto-Reconnect
  6. User kann VPN nicht manuell trennen (per Policy)

Vorteile:

  • Benutzer kann es nicht vergessen:
    • Kein "Ich habe VPN-Verbinden vergessen"
    • Gut für nicht-technische User
  • Kein manuelles Verbinden nötig:
    • One less thing to do
    • Bessere User Experience
  • Immer geschützt:
    • Auch in unsicheren Netzen (Hotel, Café)
    • Keine Sekunde ungeschützt
  • Zero Trust Architektur:
    • Gerät ist immer im "Firmennetz"
    • Continuous Authentication möglich
  • Compliance:
    • Nachweisbar: Gerät war immer geschützt
    • Audit-Log: Continuous Connection

Nachteile / Herausforderungen:

  • Keine Privatsphäre:
    • Firma sieht ALLEN Traffic (auch privat)
    • Auch YouTube, Netflix, Social Media
    • User-Akzeptanz-Problem bei BYOD
  • Kann nicht ausgeschaltet werden:
    • User kann VPN nicht deaktivieren (per Policy gesperrt)
    • Frustrierend wenn private Nutzung gewünscht
  • Performance:
    • Gesamter Traffic durch VPN → Latenz
    • Streaming (Netflix) langsamer
    • VPN-Gateway braucht viel Bandbreite
  • Single Point of Failure:
    • VPN-Gateway down = User kann nicht arbeiten
    • Auch kein privates Internet (mit Kill Switch)
    • → HA (High Availability) zwingend!
  • Geo-Restriction-Probleme:
    • User in USA, VPN-Gateway in Deutschland
    • → US-Netflix denkt User ist in DE → Content blocked

Use Cases / Wann Always-On sinnvoll:

  • Firmen-Laptops (Company-Owned):
    • Gerät gehört Firma → Firma kontrolliert
    • User-Akzeptanz höher
  • Hochsensitive Daten:
    • Healthcare (Patientendaten - HIPAA)
    • Finance (PCI DSS, SOX)
    • Government / KRITIS
  • Zero Trust Security Model:
    • Gerät muss immer authentifiziert sein
    • Continuous Security Posture Check
  • Compliance-Anforderungen:
    • Regulierung verlangt permanente Verschlüsselung

Wann Always-On NICHT geeignet:

  • BYOD (Bring Your Own Device): User-Akzeptanz gering
  • Teilzeit-Remote-Worker: User arbeitet meist im Büro
  • Nicht-sensitive Daten: Overhead nicht gerechtfertigt
  • Schlechte VPN-Infrastruktur: Ohne HA = zu riskant

Modi (Flexibilität):

  • Mandatory Always-On: User kann NICHT deaktivieren
  • Optional Always-On: User kann temporär trennen (z.B. für private Nutzung)
  • Trusted Network Detection:
    • Im Büro-WLAN → VPN auto-disconnect (nicht nötig)
    • Außerhalb Büro → VPN auto-connect
    • Guter Kompromiss!

Technologie:

  • Windows: Always On VPN (IKEv2 + Device Tunnel)
  • macOS: Always-On VPN (MDM-Profile)
  • iOS/Android: Per-App VPN oder Always-On VPN (MDM)

Best Practices:

  • ✓ Clear User-Communication (warum Always-On, was bedeutet das)
  • ✓ Betriebsrat einbeziehen (Überwachung!)
  • ✓ HA VPN-Gateway (min. 99,9% Uptime)
  • ✓ Trusted Network Detection (Auto-Disconnect im Büro)
  • ✓ Split-Tunnel wenn möglich (nicht alles durch VPN)
  • ✓ Escape-Hatch: Support kann temporär deaktivieren
Always-On = maximale Sicherheit + Komfort, aber auch maximale Kontrolle → gut kommunizieren!

Hochverfügbarkeit (HA)

Redundante VPN-Gateways für Ausfallsicherheit.

HA-Modi erklärt:

  • Active-Active:
    • Beide Gateways gleichzeitig aktiv
    • Last wird verteilt
    • Bei Ausfall: Andere übernimmt alle
    • Beste Performance
  • Active-Passive:
    • Ein Gateway aktiv, eins standby
    • Bei Ausfall: Passive wird aktiv
    • Einfacher zu konfigurieren
    • Passive Gateway "verschwendet" Ressourcen
  • Cluster:
    • Mehrere Gateways (3+) als Cluster
    • Sehr hohe Verfügbarkeit
    • Komplex zu konfigurieren
  • Keine HA:
    • Single Point of Failure
    • Gateway fällt aus = kein VPN
    • Nur für unkritische VPNs

Failover-Zeit:

Wie lange dauert Umschaltung? (meist 30 Sekunden - 2 Minuten)
HA ist Pflicht für produktive Remote-Access-VPNs!

Logging & Monitoring

Was wird protokolliert?

Logging-Level:

  • Detailliert (alle Events):
    • Jede Verbindung, jeder Traffic
    • Gut für Troubleshooting
    • ABER: Datenschutz-Problem!
    • Sehr viele Daten
  • Normal (Connect/Disconnect):
    • Wer hat sich wann verbunden?
    • Wie lange war Session?
    • Wie viel Traffic?
    • Standard-Empfehlung
  • Minimal (nur Fehler):
    • Nur fehlgeschlagene Verbindungen
    • Wenig Daten
    • Schwer zu troubleshooten

Was sollte geloggt werden:

  • Authentifizierung (erfolgreich/fehlgeschlagen)
  • Verbindungsaufbau/Trennung
  • IP-Adressen (Client + zugewiesene VPN-IP)
  • Datenvolumen
  • Fehler und Warnings

SIEM-Integration:

Logs an zentrales SIEM senden für Security-Monitoring

Datenschutz:

User haben Recht zu wissen was geloggt wird - Betriebsrat einbeziehen!
Logging ist wichtig für Security, aber Datenschutz beachten!

Notizen / VPN-Besonderheiten

Zusätzliche wichtige Informationen zum VPN.

Was dokumentieren:

  • Spezielle Firewall-Regeln (welche Ports offen?)
  • Bekannte Probleme und Workarounds
  • Split-Tunnel Konfiguration (welche Routen?)
  • ACLs und Berechtigungen (wer darf auf was zugreifen?)
  • Client-Deployment (wie verteilt?)
  • Zertifikat-Renewal-Prozess
  • Eskalations-Kontakte bei Problemen

Beispiel:

"Cisco AnyConnect 4.10 mit IKEv2/IPsec. Authentifizierung über Active Directory + Duo MFA. Split-Tunnel aktiv: nur 10.0.0.0/8 und 172.16.0.0/12 durch VPN, Internet direkt. VPN-Pool: 10.8.0.0/22 (1024 IPs). 250 Lizenzen, Peak ~180 gleichzeitige Verbindungen. HA: Active-Passive (vpn-01.example.com Primary, vpn-02.example.com Standby), Failover <60s. Posture Check: Antivirus + OS-Patches erforderlich. Bekanntes Problem: macOS Big Sur → AnyConnect 4.9 erforderlich (Upgrade läuft). Client-Deployment via SCCM/Jamf. Zertifikate: Corporate-CA, Gültigkeit 2 Jahre, Renewal 6 Monate vorher. Firewall: UDP/500 + UDP/4500 für IPsec. Logging: Connect/Disconnect Events an Splunk. Support: network-vpn@example.com, Eskalation: Network-Team-Lead bei >10 zeitgleichen Ausfällen. Compliance: DSGVO-konform (Betriebsrat informiert), Logging 90 Tage."
VPN-Dokumentation ist kritisch - bei Ausfall muss schnell reagiert werden!
