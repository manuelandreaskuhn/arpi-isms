Proxy-Server Name

Ein eindeutiger Name für Ihren Proxy-Server.

Was ist ein Proxy-Server?

Ein Vermittler zwischen Clients und dem Internet - wie ein "Dolmetscher" und "Türsteher" in einem.

Warum Proxy-Server wichtig sind:

  • Sicherheit: Zentrale Kontrolle über Internet-Zugriffe
  • Content-Filtering: Blockieren unerwünschter Websites
  • Caching: Schnellerer Zugriff durch lokale Kopien
  • Anonymität: Versteckt interne IPs nach außen
  • Bandbreiten-Kontrolle: Traffic-Management
  • Compliance: Logging für Audit-Zwecke

Namens-Beispiele:

  • "Proxy-01" - Erster Proxy-Server
  • "Corporate-Proxy" - Unternehmens-Proxy
  • "Squid-Proxy-DMZ" - Squid in der DMZ
  • "Web-Gateway" - Web Security Gateway
Moderne Unternehmen: 80%+ des Web-Traffics läuft über Proxies!

Proxy-Typ

Die Art des Proxy-Servers und welche Protokolle er verarbeitet.

Forward Proxy vs. Reverse Proxy:

  • Forward Proxy: Sitzt vor Clients (ausgehender Traffic)
  • Reverse Proxy: Sitzt vor Servern (eingehender Traffic)

Forward Proxy-Typen erklärt:

  • HTTP Proxy:
    • Nur für HTTP-Traffic (Port 80)
    • Klassischer Web-Proxy
    • Kann HTTP-Header lesen und manipulieren
    • Content-Filtering und Caching möglich
    • URL-basierte Filterung einfach
    • Heute selten allein - meist mit HTTPS kombiniert
  • HTTPS Proxy:
    • Für verschlüsselten HTTPS-Traffic (Port 443)
    • Meist mit SSL-Interception/Inspection
    • Kann verschlüsselten Traffic entschlüsseln und inspizieren
    • Braucht Proxy-CA-Zertifikat auf Clients
    • Standard heute - 90%+ des Web-Traffics ist HTTPS!
  • SOCKS4:
    • Generischer Proxy für alle TCP-Verbindungen
    • Nicht nur HTTP - auch FTP, SSH, SMTP, etc.
    • Keine Authentifizierung
    • Keine Content-Manipulation (reines Forwarding)
    • Layer 4 (Transport) Proxy
    • Veraltet - SOCKS5 bevorzugen
  • SOCKS5:
    • Erweitert SOCKS4 um:
    • UDP-Support (nicht nur TCP)
    • Authentifizierung (Username/Password)
    • IPv6-Support
    • Für Apps die keinen HTTP-Proxy unterstützen
    • Gut für SSH-Tunneling, Torrents
    • Flexibler als HTTP-Proxy, aber weniger Kontrolle
  • Transparent Proxy (Intercepting Proxy):
    • Client weiß nichts vom Proxy
    • Traffic wird auf Netzwerk-Ebene umgeleitet (via Firewall/Router)
    • Keine Browser-Konfiguration nötig
    • Port 80/443 Traffic wird automatisch abgefangen
    • Gut für erzwungenes Filtering
    • Problem: Kann HTTPS-Apps verwirren (Zertifikats-Pinning)
    • Praktisch, aber kann Kompatibilitätsprobleme verursachen

Reverse Proxy (spezieller Typ):

  • Sitzt VOR Webservern (nicht vor Clients!)
  • Clients verbinden sich zum Reverse Proxy
  • Reverse Proxy leitet an Backend-Webserver weiter
  • Einsatzzwecke:
    • Load Balancing (Traffic auf mehrere Server verteilen)
    • SSL-Terminierung (HTTPS entschlüsseln, HTTP zu Backend)
    • Caching (statische Inhalte cachen)
    • Schutz (Webserver nicht direkt erreichbar)
    • WAF (Web Application Firewall) Integration
    • Kompression (Gzip/Brotli)
  • Typische Software: Nginx, HAProxy, Apache mod_proxy
  • Komplett anderer Use Case als Forward Proxy!

Welcher Typ für was?

  • Unternehmens-Internet-Zugang: HTTP/HTTPS Proxy
  • App-Support (nicht HTTP): SOCKS5
  • Erzwungenes Filtering: Transparent Proxy
  • Webserver schützen: Reverse Proxy
HTTP/HTTPS Forward Proxy ist Standard für Unternehmens-Internet-Zugang!

Proxy-Port

Der TCP-Port, auf dem der Proxy lauscht.

Typische Proxy-Ports:

  • 3128: Standard Squid Proxy Port - am häufigsten
  • 8080: Alternative, oft für HTTP-Proxies
  • 8888: Weitere Alternative
  • 80: Transparent Proxy (Standard HTTP-Port)
  • 443: Für HTTPS-Proxies
  • 1080: Standard SOCKS-Port

Wichtig:

Port muss in Firewall geöffnet sein!

Proxy-Software

Welche Software wird als Proxy verwendet?

Open Source Forward Proxies:

  • Squid Proxy:
    • Der Klassiker - seit 1996
    • HTTP/HTTPS Caching Proxy
    • Läuft auf Linux/Unix/BSD
    • Sehr flexibel konfigurierbar (aber Lernkurve!)
    • ACLs für komplexe Zugriffsregeln
    • SSL-Bump für HTTPS-Inspection
    • Integrierbar mit ICAP für Antivirus
    • Standard in vielen Unternehmen - kostenlos, stabil, bewährt
  • Privoxy:
    • Privacy-fokussierter Proxy
    • Filtert Werbung und Tracking
    • Für einzelne User oder kleine Gruppen
  • Tinyproxy:
    • Minimalistisch, sehr klein
    • Für einfache Szenarien

Open Source Reverse Proxies:

  • Nginx:
    • Moderner, sehr performanter Reverse Proxy
    • Auch Load Balancer und Webserver
    • Geringer Speicher-Verbrauch
    • Event-driven (nicht prozess-basiert wie Apache)
    • Gut für hohe Konkurrenz (viele gleichzeitige Connections)
    • SSL-Terminierung, HTTP/2, HTTP/3 Support
    • De-facto Standard für moderne Web-Apps
  • HAProxy:
    • High-Performance Load Balancer und Reverse Proxy
    • Fokus auf Reliability und Performance
    • TCP und HTTP Load Balancing
    • Sehr schnell, sehr stabil
    • Umfangreiches Health-Checking
    • Wenn Performance kritisch ist - HAProxy wählen
  • Apache mod_proxy:
    • Apache Webserver als Reverse Proxy
    • Wenn Apache schon vorhanden
    • Weniger performant als Nginx/HAProxy
  • Traefik:
    • Modern, Cloud-Native Reverse Proxy
    • Automatische Service-Discovery
    • Perfekt für Docker/Kubernetes
    • Let's Encrypt Integration

Enterprise / Commercial Proxies:

  • Blue Coat ProxySG (Symantec, jetzt Broadcom):
    • Enterprise Secure Web Gateway (SWG)
    • Hardware-Appliance oder Virtual
    • All-in-One: Proxy + Firewall + DLP + Antivirus + URL-Filter
    • Sehr leistungsstark (10.000+ User)
    • Sehr teuer (€50k-200k+)
    • Rolls-Royce der Proxies - für Großunternehmen
  • Fortigate Web Proxy:
    • Teil der Fortigate UTM-Firewall
    • Integriert mit anderen Security-Features
    • Gutes Preis-/Leistungsverhältnis
    • Für KMU bis Mittelstand
  • Cisco WSA (Web Security Appliance):
    • Cisco's Secure Web Gateway
    • Advanced Threat Protection
    • Integration mit Cisco Security-Ecosystem
    • Teuer, Enterprise-fokussiert
  • McAfee Web Gateway:
    • Teil der McAfee Security-Suite
    • DLP, Malware-Scanning, URL-Filtering
  • Palo Alto Networks:
    • Next-Gen Firewall mit Proxy-Funktionen
    • App-ID, User-ID, Content-ID
    • Sehr umfangreich, komplex

Cloud-basierte Proxies (SaaS):

  • Zscaler:
    • Cloud-basierter Secure Web Gateway
    • Kein eigener Server nötig
    • Clients verbinden zu Zscaler-Cloud
    • Gut für verteilte Standorte und Home-Office
    • Skaliert automatisch
    • Subscription-Modell (pro User)
    • Zukunft - Zero Trust Network Access (ZTNA)
  • Cloudflare Gateway:
    • Cloud-Proxy von Cloudflare
    • Integriert mit Cloudflare's globalem Netzwerk
    • Schnell durch weltweite Edge-Location
  • Cisco Umbrella (früher OpenDNS):
    • DNS-basierter Cloud-Proxy
    • Einfach zu deployen

Entscheidungskriterien:

  • Budget: Open Source (Squid) vs. Commercial (Blue Coat €€€)
  • User-Anzahl: Squid bis ~5000, Enterprise-Lösung 10k+
  • Features: Nur Proxy oder auch DLP, Malware-Scan?
  • Expertise: Linux-Admin für Squid, oder Appliance bevorzugt?
  • Cloud vs. On-Prem: Viele Remote-User → Cloud (Zscaler)
  • Compliance: Spezielle Anforderungen (KRITIS)?

Typische Kombinationen:

  • KMU (bis 100 User): Squid oder Fortigate
  • Mittelstand (100-1000): Squid oder Cisco WSA
  • Enterprise (1000+): Blue Coat, Zscaler
  • Modern/Cloud: Zscaler, Cloudflare Gateway
Squid ist kostenlos und mächtig - aber braucht Linux-Expertise!

Authentifizierung

Wie authentifizieren sich Benutzer am Proxy?

Warum Authentifizierung wichtig:

  • Accountability: Wer hat was im Web gemacht? (Audit-Trail)
  • Granulare Regeln: Chef darf alles, Praktikant nur Whitelist
  • Reporting: Traffic-Statistiken pro User/Gruppe
  • Compliance: Gesetzliche Anforderungen (wer hat was heruntergeladen)
  • Incident Response: Bei Security-Vorfall: Welcher User?

Authentifizierungs-Methoden im Detail:

  • Keine (IP-basiert):
    • Jeder im erlaubten Netz kann nutzen
    • Identifikation nur via IP-Adresse
    • Kein User-Tracking möglich
    • Einfach, aber unsicher und nicht compliant
    • Nur für Test-Umgebungen oder Gast-Netze
  • Basic Authentication:
    • Browser fragt nach Username + Passwort
    • Credentials als Base64 übertragen (NICHT verschlüsselt!)
    • Kann abgefangen werden (Man-in-the-Middle)
    • Nur zusammen mit HTTPS verwenden!
    • User muss bei jedem Browser-Start eingeben
    • Veraltet - nicht empfohlen
  • NTLM (NT LAN Manager):
    • Microsoft-Authentifizierung (Challenge-Response)
    • Transparent für Windows-Clients (Automatic Windows Logon)
    • User merkt nichts - verwendet Windows-Anmeldedaten
    • Authentifizierung gegen Active Directory
    • NTLM v1 unsicher (veraltet), NTLMv2 okay
    • Problem: Mehrere Roundtrips nötig (Latenz)
    • Standard in Windows-Umgebungen - einfach für User
  • Kerberos:
    • Moderner Windows-Auth-Standard (seit Windows 2000)
    • Sicherer als NTLM (Mutual Authentication)
    • Single Sign-On möglich (ein Login für alle Dienste)
    • Ticket-basiert (nicht Passwort-basiert)
    • Braucht korrekte Zeit-Synchronisation (NTP wichtig!)
    • Komplexer zu konfigurieren als NTLM
    • Best Practice für Windows-Umgebungen
  • Digest Authentication:
    • Sicherer als Basic (Passwort wird gehashed)
    • Challenge-Response-Mechanismus
    • Selten verwendet (NTLM/Kerberos besser)
  • LDAP (Lightweight Directory Access Protocol):
    • Authentifizierung gegen LDAP-Server
    • Plattformübergreifend (Windows, Linux, macOS)
    • Flexibel - auch mit OpenLDAP, FreeIPA
    • User gibt Credentials manuell ein
    • Kann mit NTLM/Kerberos kombiniert werden
  • RADIUS:
    • Zentrale Auth-Server (wie ISPs nutzen)
    • Oft für Network Access Control
    • Kann mit 2FA integriert werden (Token, SMS)
    • Gut für Multi-Faktor-Authentifizierung
  • SAML / OAuth / OpenID Connect:
    • Moderne Web-basierte SSO-Standards
    • Für Cloud-Proxies (Zscaler, Cloudflare)
    • Integriert mit Identity Provider (Okta, Azure AD)

Typische Konfiguration (Windows-Umgebung):

  1. Primary: Kerberos (Automatic Windows Logon)
  2. Fallback: NTLM (wenn Kerberos fehlschlägt)
  3. Fallback: Basic over HTTPS (für mobile Geräte)

Authentication Backend:

Wo werden User/Passwörter geprüft?
  • Active Directory: Standard in Windows-Umgebungen
  • LDAP: OpenLDAP, FreeIPA, etc.
  • Local File: Proxy-eigene User-Datenbank (nur für Test)
  • External Helper: Custom Script (z.B. gegen MySQL-DB)

Best Practices:

  • Kerberos + NTLM Fallback für Windows-Umgebungen
  • Transparent Auth (keine User-Interaktion)
  • Gruppenbasierte ACLs (AD-Gruppen → Proxy-Regeln)
  • Logging von Username (nicht nur IP)
NTLM/Kerberos ist Standard - transparent für Windows-User!

Proxy-Ausnahmen (No Proxy / Bypass-Liste)

Welche Hosts/Netze umgehen den Proxy?

Warum Ausnahmen wichtig:

  • Interne Server sollen nicht über Proxy (Performance)
  • Localhost immer ausgenommen
  • Private IP-Bereiche meist direkt erreichbar

Typische Ausnahmen:

  • localhost, 127.0.0.1: Lokale Adresse
  • *.local: Lokale Domain
  • 10.0.0.0/8: Privates Netz (RFC 1918)
  • 172.16.0.0/12: Privates Netz
  • 192.168.0.0/16: Privates Netz
  • internal.example.com: Intranet-Server

Syntax-Varianten:

  • Komma-getrennt: "localhost, 127.0.0.1, *.local"
  • Zeilenweise: Eine Ausnahme pro Zeile
  • Wildcard: *.example.com (alle Subdomains)
  • CIDR: 10.0.0.0/8 (ganzes Netz)
Zu viele Ausnahmen = Proxy wird umgangen = weniger Sicherheit!

PAC-Datei (Proxy Auto-Config)

JavaScript-Datei für automatische Proxy-Konfiguration.

Was ist PAC?

Eine JavaScript-Datei, die dem Browser sagt: "Für diese URL nimm Proxy X, für jene URL direkt".

Warum PAC verwenden:

  • Komplexe Routing-Logik (verschiedene Proxies für verschiedene Ziele)
  • Automatische Proxy-Auswahl
  • Zentrale Verwaltung (Änderungen in PAC, nicht auf jedem Client)
  • Load Balancing über mehrere Proxies

Beispiel PAC-URL:

  • http://proxy.example.com/proxy.pac
  • http://wpad.example.com/wpad.dat

PAC-Beispiel-Code:

function FindProxyForURL(url, host) {
    if (isInNet(host, "10.0.0.0", "255.0.0.0"))
        return "DIRECT";
    else
        return "PROXY proxy.example.com:3128";
}
PAC ist mächtig, aber braucht JavaScript-Kenntnisse!

WPAD (Web Proxy Auto-Discovery)

Automatische Erkennung der Proxy-Konfiguration.

Was ist WPAD?

Browser sucht automatisch nach PAC-Datei - ohne manuelle Konfiguration!

Wie WPAD funktioniert:

  1. Browser sucht DHCP-Option 252 (WPAD-URL)
  2. Falls nicht gefunden: DNS-Abfrage "wpad.domain.com"
  3. Browser lädt http://wpad.domain.com/wpad.dat
  4. PAC-Datei wird ausgeführt

Vorteile:

  • Keine manuelle Proxy-Konfiguration nötig
  • Funktioniert automatisch
  • Gut für mobile Clients (Laptop wechselt Standort)

Risiken:

  • WPAD-Hijacking möglich (Angreifer fälscht wpad.dat)
  • Sicherheits-Best-Practice: WPAD deaktivieren oder absichern
WPAD ist praktisch, aber Sicherheitsrisiko - gut überlegen!

SSL-Interception / HTTPS-Inspection

Entschlüsselung von HTTPS-Traffic zur Inspektion.

Das fundamentale Problem:

  • Heute ist 90%+ des Web-Traffics HTTPS (verschlüsselt)
  • Proxy kann Inhalt nicht sehen (nur Domain, keine URL/Content)
  • Malware versteckt sich in HTTPS
  • DLP funktioniert nicht bei HTTPS
  • Content-Filtering limitiert

Was ist SSL-Interception / HTTPS-Inspection / SSL-Bump?

Proxy agiert als kontrollierter Man-in-the-Middle:

Technischer Ablauf:

  1. Client → Proxy:
    • Client will zu https://example.com
    • TLS-Handshake mit Proxy (Client denkt es ist example.com)
    • Proxy präsentiert Fake-Zertifikat "example.com"
    • Signiert mit Proxy-CA (muss auf Client installiert sein!)
  2. Proxy entschlüsselt:
    • Proxy hat nun Klartext (HTTP)
    • Kann inspizieren: URL, Content, Malware-Scan, DLP
  3. Proxy → Server:
    • Proxy baut separate HTTPS-Verbindung zur echten example.com
    • Mit echtem Zertifikat von example.com
  4. Traffic fließt:
    • Proxy entschlüsselt (Client → Proxy)
    • Proxy inspiziert
    • Proxy verschlüsselt neu (Proxy → Server)

Modi / Konfigurationen:

  • Aktiviert (Full Inspection):
    • Alle HTTPS-Verbindungen werden intercepted
    • Maximum Security, aber auch Maximum Privacy-Eingriff
  • Teilweise (Blacklist/Whitelist):
    • Whitelist: Banking, Gesundheits-Portale NICHT intercepten (Privacy)
    • Blacklist: Nur verdächtige Sites intercepten
    • Kompromiss Security vs. Privacy
    • Praktischer Ansatz - empfohlen
  • Deaktiviert:
    • Kein Blick in HTTPS (nur Domain sichtbar)
    • Malware kann sich verstecken
    • DLP funktioniert nicht
    • Content-Filtering sehr limitiert

Voraussetzungen:

  • Proxy-CA-Zertifikat auf allen Clients installieren:
    • Via GPO (Group Policy) in Windows-Umgebung
    • Oder MDM (Mobile Device Management) für Smartphones
    • Sonst: Zertifikatsfehler bei jeder HTTPS-Seite!
  • Leistungsfähiger Proxy: SSL-Handshakes brauchen CPU

Probleme / Challenges:

  • Certificate Pinning:
    • Apps die nur bestimmte Zertifikate akzeptieren
    • Funktionieren mit Interception NICHT (z.B. Banking-Apps)
    • → Müssen auf Whitelist
  • Performance:
    • SSL-Handshake ist rechenintensiv
    • Proxy braucht viel CPU
    • Hardware-Beschleunigung (SSL-Offload-Karten) nötig bei 1000+ User
  • Perfect Forward Secrecy (PFS):
    • Moderne Cipher-Suites mit PFS
    • Proxy muss unterstützen

Rechtliche / Datenschutz-Aspekte:

  • DSGVO: Tiefer Eingriff in Privatsphäre
  • Betriebsrat: Muss zustimmen (in Deutschland Pflicht)
  • Betriebsvereinbarung: Regelt was überwacht wird
  • Mitarbeiter informieren: Transparenz erforderlich
  • Whitelist für Private Sites: Banking, Gesundheit, Gewerkschaften

Empfehlung - Balanced Approach:

  • ✓ Interception aktiviert für Security (Malware, Phishing)
  • ✓ Whitelist für Privacy-sensitive Sites (Banking, Health)
  • ✓ Betriebsvereinbarung abschließen
  • ✓ Mitarbeiter informieren (Awareness)
  • ✓ Logging auf Security-Incidents beschränken (nicht Surf-Gewohnheiten)
SSL-Interception ist das größte Privacy vs. Security Dilemma - gut abwägen!

Proxy-Cache

Lokales Speichern von Web-Inhalten zur Beschleunigung und Bandbreiten-Einsparung.

Was ist Caching?

Proxy speichert heruntergeladene Inhalte lokal - bei erneutem Abruf: sofort aus lokalem Cache statt erneut Download!

Wie Caching funktioniert:

  1. User A ruft example.com/logo.png ab
  2. Proxy lädt logo.png herunter, liefert an User A
  3. Proxy speichert logo.png in Cache
  4. User B ruft example.com/logo.png ab
  5. Proxy liefert aus Cache (Cache HIT) - kein erneuter Download
  6. Schneller + spart Internet-Bandbreite

Was wird gecacht:

  • Statische Inhalte:
    • Bilder (JPG, PNG, GIF, SVG)
    • CSS-Dateien (Stylesheets)
    • JavaScript-Dateien
    • Fonts (WOFF, TTF)
  • Downloads:
    • Software-Updates (Windows Updates, Antivirus-Updates)
    • PDFs, Office-Dokumente
    • Installationsdateien
  • Streaming (teilweise):
    • YouTube-Videos (bei mehrfachem Abruf)
    • Aber: Zunehmend schwierig wegen HTTPS

Was NICHT gecacht wird:

  • Dynamische Seiten (immer neu generiert)
  • Login-Bereiche (personalisiert)
  • HTTPS-Seiten (ohne SSL-Interception)
  • Seiten mit "Cache-Control: no-cache" Header
  • POST-Requests (nur GET wird gecacht)

Cache-Größe:

  • Hängt ab von:
    • Anzahl Benutzer
    • Traffic-Volumen
    • Wie lange Cache behalten (Cache-Retention)
  • Faustregel:
    • 50-100 User: 10-20 GB
    • 100-500 User: 20-50 GB
    • 500-1000 User: 50-100 GB
    • 1000+ User: 100-500 GB
  • Mehr Cache = höhere HIT-Rate, aber mehr Festplatte nötig
  • SSD bevorzugen (schneller als HDD)

Cache-Strategien:

  • LRU (Least Recently Used): Älteste Inhalte werden zuerst gelöscht
  • LFU (Least Frequently Used): Selten abgerufene Inhalte zuerst löschen
  • FIFO (First In First Out): Einfach, nicht optimal

Cache Hierarchien (bei mehreren Proxies):

  • Parent Proxy (zentraler Cache)
  • Child Proxies (pro Standort)
  • Child fragt Parent bei Cache MISS
  • Spart WAN-Bandbreite

Cache-Vorteile:

  • ✓ Schnellerer Seitenabruf (aus lokalem Cache)
  • ✓ Weniger Internet-Bandbreite (große Einsparungen möglich)
  • ✓ Entlastung der Original-Webserver
  • ✓ Kosten-Einsparung (Traffic-Kosten)

Warum Caching heute weniger wichtig:

  • HTTPS dominiert:
    • Ohne SSL-Interception kein Caching möglich
    • Mit SSL-Interception: Privacy-Bedenken
  • Schnelles Internet:
    • Bandbreite kein Engpass mehr (Glasfaser, 1Gbit+)
    • Latenz zu Cloud-Servern niedrig (CDNs)
  • CDNs:
    • Content Delivery Networks (Cloudflare, Akamai)
    • Caching schon auf Server-Seite
  • Dynamische Inhalte:
    • Mehr personalisierter Content (nicht cachbar)

Wann Caching noch sinnvoll:

  • Langsame/teure Internet-Anbindung (Satellit, teures Volumen)
  • Viele User rufen gleiche Inhalte ab (Intranet, Update-Server)
  • Software-Updates cachen (Windows, Antivirus)

Moderne Alternative:

Explizite Caching-Server:
  • Windows WSUS (Windows Update Server) - cachet Windows-Updates
  • Steam Cache (für Gaming)
  • Specialized Caching für spezifische Dienste
Caching war wichtig in den 2000ern - heute weniger relevant (HTTPS, schnelles Internet, CDNs)

Failover / Sekundärer Proxy

Redundanter Proxy für Hochverfügbarkeit.

Warum Failover wichtig:

Wenn Proxy ausfällt → kein Internet mehr!

Failover-Szenarien:

  • Automatisch:
    • Browser probiert automatisch nächsten Proxy
    • Via PAC-Datei: "PROXY proxy1:3128; PROXY proxy2:3128"
    • Seamless für Benutzer
  • Manuell:
    • Bei Ausfall: Admin ändert Konfiguration
    • Downtime während Umstellung
  • Load Balanced:
    • Traffic verteilt auf beide Proxies
    • Bessere Performance + Redundanz
    • Via DNS Round-Robin oder echter Load Balancer

Best Practice:

Mindestens 2 Proxies für Produktiv-Umgebung

URL-Filtering

Blockieren bestimmter Websites nach Kategorien.

Was ist URL-Filtering:

Proxy prüft aufgerufene URLs gegen Datenbank und blockiert nach Regeln.

Typische blockierte Kategorien:

  • Adult Content / Pornographie
  • Malware & Phishing
  • Gambling / Glücksspiel
  • Social Media (optional)
  • Streaming (optional - Netflix, YouTube)
  • File Sharing (Dropbox, WeTransfer)
  • Anonymizer / VPN-Services

URL-Datenbanken:

  • Kommerzielle Dienste (Brightcloud, WebPulse)
  • Werden ständig aktualisiert (neue Malware-Sites)
  • Oft Subscription-basiert

Use Cases:

  • Sicherheit: Malware-Sites blockieren
  • Compliance: Illegale Inhalte verhindern
  • Produktivität: Zeitfresser blockieren (umstritten)
URL-Filtering = zentraler Schutz vor Web-Threats!

Malware-Scanning

Scannt Downloads auf Schadsoftware.

Was macht Malware-Scanning:

Proxy scannt heruntergeladene Dateien auf Viren und Malware – bevor sie auf Client-PC landen!

Was wird gescannt:

  • Downloads (EXE, ZIP, Office-Dokumente)
  • E-Mail-Anhänge (wenn Webmail über Proxy)
  • Eingebettete Dateien in Webseiten

Scan-Methoden:

  • Signatur-basiert (bekannte Malware)
  • Heuristik (verdächtiges Verhalten)
  • Sandboxing (Datei in isolierter Umgebung ausführen)
  • Cloud-Reputation (File-Hash gegen Datenbank)

Herausforderung:

Performance – große Dateien scannen dauert!
Zusätzlicher Schutz zur Endpoint-Security!

DLP (Data Loss Prevention)

Verhindert unautorisierten Abfluss sensibler Unternehmensdaten.

Was ist DLP?

System das ausgehenden Traffic überwacht und sensible Daten erkennt und blockiert bevor sie das Unternehmen verlassen.

Das Problem (ohne DLP):

  • Mitarbeiter (absichtlich oder versehentlich) sendet:
    • Kundenliste an Privat-E-Mail
    • Geschäftsgeheimnis zu Konkurrenz
    • Quellcode auf GitHub
    • Patientendaten via Webmail
  • → Data Breach, Compliance-Verstoß, Imageschaden

Was DLP erkennt (Detection-Methoden):

  • Pattern Matching / Regular Expressions:
    • Kreditkartennummern (16 Ziffern, Luhn-Check)
    • Sozialversicherungsnummern
    • IBANs (DE + 20 Ziffern)
    • E-Mail-Adressen (@company.com)
  • Keyword-basiert:
    • "Vertraulich", "Geheim", "Confidential"
    • Projektcode-Namen
    • Kombination von Keywords
  • Fingerprinting / Hashing:
    • DLP lernt "Fingerabdruck" sensibler Dokumente
    • Erkennt auch modifizierte Versionen
  • Exact Data Match (EDM):
    • Vergleich mit exakter Liste (z.B. alle Kundennummern)
    • Hash-basiert (Privacy-preserving)
  • Document Matching:
    • Erkennt firmen-interne Dokumente (Watermarks, Metadaten)
  • Optical Character Recognition (OCR):
    • Scannt Text in Bildern/Screenshots
    • User macht Screenshot von Kundenliste → DLP erkennt
  • Machine Learning:
    • Lernt was "sensibel" aussieht
    • Klassifizierung von Dokumenten

DLP-Aktionen:

  • Block:
    • Upload/E-Mail wird verhindert
    • User sieht Fehlermeldung
    • Für kritische Daten (Patientendaten, Finanzdaten)
  • Warn & Allow:
    • User wird gewarnt, kann aber fortfahren
    • Für mittlere Sensitivität
    • User muss Grund angeben
  • Justify:
    • User muss Business-Rechtfertigung eingeben
    • Manager-Genehmigung erforderlich
  • Log Only (Monitor Mode):
    • Erlauben, aber Incident loggen
    • Für Testing-Phase
    • Audit-Trail
  • Quarantine:
    • Datei wird isoliert
    • Security-Team prüft manuell
  • Encrypt:
    • Automatische Verschlüsselung sensibler E-Mails

DLP-Kanäle (wo wird überwacht):

  • Web Proxy: HTTP/HTTPS Uploads (Webmail, Cloud-Storage)
  • E-Mail Gateway: Ausgehende E-Mails + Anhänge
  • Endpoint DLP: Agent auf Client-PCs (USB, Drucker, Screenshots)
  • Network DLP: Alle Netzwerk-Traffic (FTP, etc.)
  • Cloud DLP: SaaS-Anwendungen (Office 365, Google Drive)

Use Cases:

  • Compliance:
    • DSGVO: Schutz personenbezogener Daten
    • HIPAA: Patientendaten dürfen nicht abfließen
    • PCI-DSS: Kreditkartendaten schützen
    • SOX: Finanzdaten
  • Intellectual Property (IP) Schutz:
    • Quellcode nicht zu GitHub
    • CAD-Zeichnungen nicht zu Privat-Mail
    • Patente, Forschungsergebnisse
  • Insider Threat:
    • Mitarbeiter der kündigt und Daten mitnehmen will
    • Sabotage

Herausforderungen:

  • False Positives:
    • Legitime Daten werden fälschlich als sensibel erkannt
    • User-Frustration ("Ich kann nicht arbeiten!")
    • → Sorgfältiges Tuning nötig
  • Performance:
    • Deep Content Inspection ist rechenintensiv
    • Große Dateien scannen dauert
  • Verschlüsselung:
    • HTTPS: Braucht SSL-Interception
    • End-to-End verschlüsselte Messenger: DLP blind
  • Management-Overhead:
    • Policies pflegen
    • False Positives behandeln
    • User-Schulung

Best Practices:

  • Start mit "Monitor Mode" (nur loggen, nicht blockieren)
  • Analyse: Was sind echte Risiken, was False Positives?
  • Policies schrittweise verschärfen
  • User-Awareness-Training (Warum DLP wichtig ist)
  • Clear Escalation Path (User kann Freigabe anfragen)

DLP-Lösungen:

  • Symantec DLP
  • Forcepoint DLP
  • McAfee DLP
  • Microsoft Purview (in Office 365)
  • Digital Guardian
DLP ist komplex und braucht sorgfältiges Tuning - aber essentiell für Compliance!

Access Logging

Protokollierung der Proxy-Zugriffe.

Was wird geloggt:

  • Wer (Benutzer oder IP)
  • Wann (Zeitstempel)
  • Wohin (URL)
  • Was (HTTP-Methode, Status-Code)
  • Wie viel (Bytes übertragen)

Log-Level:

  • Vollständig: Alle Requests inkl. URLs – detailliert, aber viel Daten
  • Minimal: Nur IP, Zeit, Domain (nicht volle URL) – weniger Daten
  • Deaktiviert: Kein Logging – nicht empfohlen!

Warum Logging wichtig:

  • Security: Angriffe erkennen
  • Troubleshooting: Warum wurde Site blockiert?
  • Compliance: Audit-Trail
  • Forensik: Bei Incident zurückverfolgen

Datenschutz:

  • Persönliche Surf-Gewohnheiten sind sensibel
  • Log-Retention begrenzen (90 Tage typisch)
  • Zugriff beschränken
  • Betriebsrat einbeziehen

Log-Größe:

Bei vielen Benutzern: GB pro Tag!
Logging ist Pflicht für Security, aber Datenschutz beachten!

Notizen / Proxy-Besonderheiten

Zusätzliche wichtige Informationen zum Proxy-Server.

Was dokumentieren:

  • Spezielle ACL-Regeln (wer darf was)
  • Whitelist-Einträge (welche Sites erlaubt trotz Filterung)
  • Blacklist-Einträge (manuell geblockte Sites)
  • Performance-Tuning (Cache-Einstellungen)
  • Bekannte Probleme und Workarounds
  • Change-Prozesse (wie werden Änderungen durchgeführt)
  • Eskalations-Kontakte

Beispiel:

"Squid 5.7 auf Ubuntu 22.04. SSL-Interception aktiviert mit Corporate-CA (installiert via GPO). URL-Filtering: Brightcloud-Kategorien 'Adult', 'Malware', 'Phishing' blockiert. Social Media teilweise erlaubt (LinkedIn ja, Facebook/Instagram nein). Whitelist: *.microsoft.com, *.office365.com (wegen Teams-Probleme). Bekanntes Problem: YouTube-Videos laden langsam bei >500 gleichzeitigen Usern → Cache-Optimization geplant. PAC-Datei: http://proxy.example.com/proxy.pac (via WPAD). Failover zu Proxy-02 (proxy-02.example.com:3128) via PAC. Support: network-team@example.com, Eskalation: CTO bei Ausfall >1h."
Gute Proxy-Dokumentation spart Stunden bei Troubleshooting!
